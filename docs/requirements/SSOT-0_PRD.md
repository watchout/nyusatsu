## 1. プロダクト概要

### 1.1 ビジョン

```
入札版の freee ＋ オートパイロット。
官公庁入札の案件を自動で探し、仕様書をAIが読解し、
応札に必要な準備を半自動で進める。
自分で手を動かさなくても、元請けとして入札事業が回る仕組み。
```

### 1.2 ポジショニング

```
カテゴリ: 入札オペレーションOS

NJSS/入札王 = 「見つける（探索）」が主戦場
ホットプラス = 「教える・伴走する（コンサル）」が主戦場
入札ラクダAI = 「進める（実務の半自動化）」が主戦場

→ 「入札情報サービス」でも「コンサルサービス」でもない第3のカテゴリ
→ NJSSの後工程を奪う/補完し、ホットプラスのノウハウをツール化して代替する
```

### 1.3 解決する課題（一言）

```
入札は儲かるのに、探す・読む・手続きの手間が膨大すぎて、
本業が忙しい経営者が参入・継続できない。
```

## 3. 課題と価値提案

> 詳細: docs/idea/VALUE_PROPOSITION.md

### 3.1 6地獄（顧客の痛み）

| # | 痛み | 深刻度 | MVP対応 |
|---|------|-------|--------|
| 1 | 探索地獄 — 案件が分散、おいしい案件を探しきれない | 高 | ✅ |
| 2 | 読解地獄 — 用語・条件がわからず判断できない | 高 | ✅ |
| 3 | 手続き地獄 — 提出漏れ・形式不備・期限切れのリスク | 高 | ✅ |
| 4 | 見積地獄 — 入札価格の相場がわからない | 中 | ✅ |
| 5 | 結果地獄 — 負けた理由が残らず次に活かせない | 中 | △（記録まで） |
| 6 | 時間地獄 — 本業が忙しく入札に手が回らない | 高 | ✅ |

### 3.2 価値提案ステートメント

```
統一資格を取ったが入札経験がなく、
本業が忙しくて入札に手が回らない中小企業経営者が

「入札を事業として自動で回したい」と思っているとき、

入札ラクダAIは
案件探索→仕様書理解→応札準備を半自動化し、
毎朝10分の案件チェックだけで入札事業が回る仕組みを提供します。

NJSSや入札王（案件を探すところまで）と違って、
「探した後」の実務（読む→判断する→準備する→学習する）まで
一気通貫で半自動化する「入札オペレーションOS」です。

ホットプラス（人手のコンサル）と違って、
ノウハウをAIに吸収させたツールとして提供するため、
コンサルに頼らず自分のペースで入札を回せます。
```

### 3.3 毎朝の具体シーン

> 詳細: VALUE_PROPOSITION.md §3.2

```
07:00 — ダッシュボード確認（1分）  → 新着3件のスコア付きカード表示
07:02 — おすすめ1件を「応札予定」に（3分） → AI読解カード確認＋ボタン1つ
07:05 — チェックリスト＋価格分析を確認（4分） → 準備タスク＋相場が見える
07:09 — 残り2件を「見送り」（1分） → 理由入力して終了
07:10 — 本業へ。合計10分。判断だけ。探索・読解・準備はAIが済ませている。
```

### 3.4 時短の定義

> 詳細: VALUE_PROPOSITION.md §3.3

| # | 工程 | Before（手作業） | After（入札ラクダAI） | 削減率 |
|---|------|-----------------|---------------------|--------|
| 1 | 案件探索 | 60〜120分/日 | 0分（自動収集） | 100% |
| 2 | 案件スクリーニング | 30分/日 | 3分 | 90% |
| 3 | 公告・仕様書の読解 | 2〜4時間/件 | 2分/件 | 98% |
| 4 | 参加可否の判断 | 30分/件 | 0分（自動判定） | 100% |
| 5 | 提出物リスト作成 | 1時間/件 | 0分（自動生成） | 100% |
| 6 | スケジュール作成 | 30分/件 | 0分（自動生成） | 100% |
| 7 | 価格調査 | 1〜2時間/件 | 2分/件 | 95% |

**合計（1案件あたり）**: Before 5〜9時間 → After **7分** ※1

> ※1 前提: F-001完了（公告取得済み）、F-002完了（AI読解済み）、company_profile登録済み。「日次運用の判断時間」であり、初回セットアップ時間は含まない。

### 3.5 不備ゼロの定義

> 詳細: VALUE_PROPOSITION.md §3.4

| # | ゼロにする不備 | 防止策 |
|---|-------------|-------|
| 1 | 提出書類の漏れによる無効入札 | AI抽出→チェックリスト化。未チェックなら「未完了」表示 |
| 2 | 参加資格の不適合による失格 | 5条件自動判定（統一資格・等級・品目・地域・免許） |
| 3 | 提出期限超過による無効 | 逆算スケジュール自動生成＋期限切れ自動アーカイブ |
| 4 | 参加条件の誤読による応札判断ミス | AI読解に根拠箇所（ページ＋原文リンク）を付与 |
| 5 | 価格データ未確認による大幅な価格乖離 | 類似案件の落札実績を自動表示 |

**検証方法**: Phase1の全入札実績で計測。目標: 初回入札から6ヶ月間、上記5項目すべてゼロ。

### 3.6 フィットスコア

| 指標 | スコア | 判定 |
|------|-------|------|
| 重要な痛みへの対応率 | 5/6 | 良好 |
| 重要な利得への対応率 | 4/5 | 良好 |
| 総合フィット度 | 9/10 | **Go** |

### 4.2 機能詳細（P0）

> 各P0機能の詳細仕様は、SSOT-1_FEATURE_CATALOG.md 以降で定義する。
> ここでは PRD レベルの要件を記載する。

#### F-001: 案件自動収集＋AIフィルタリング

| 項目 | 内容 |
|------|------|
| **入力** | 検索条件（キーワード、対象領域、地域、等級） |
| **処理** | ① 対象サイトを定期巡回しHTML/PDF/CSVを取得 ② 案件情報を正規化 ③ キーワード＋条件でフィルタ ④ スコアリング |
| **出力** | スコア付き案件カード一覧（案件名、発注機関、期限、スコア、ステータス） |
| **対象データソース** | 調達ポータル（落札実績OD）、公開入札サイト（段階的に追加） |
| **対象領域（Phase1）** | ① 軽運送・配送 ② 物品調達（物販） ③ 内装・雑務系役務 |
| **更新頻度** | 1日1回以上（差分検知） |
| **制約** | データ取得失敗時のリトライ・アラート必須。原本（HTML/PDF/CSV）を保存 |

#### F-002: 公告・仕様書のAI読解＋案件カード化

| 項目 | 内容 |
|------|------|
| **入力** | 公告HTML/PDF、仕様書PDF |
| **処理** | ① LLMで公告・仕様書を読解 ② 要点を構造化抽出（参加条件、提出物、期限、リスク、業務内容） |
| **出力** | 案件カード（構造化された要点＋根拠箇所＋原文リンク） |
| **AI読解の原則** | 抽出結果は断定しない。必ず根拠箇所（ページ・見出し・引用）を保持。最終判断は人間に委ねる |
| **知識ベース** | ホットプラスのノウハウ（KNOWLEDGE_DIGEST.md）を参照し、用語解説・判断基準を補完 |

#### F-003: 参加可否の自動判定

| 項目 | 内容 |
|------|------|
| **入力** | 案件カード（F-002の出力）＋ユーザーの資格情報 |
| **処理** | ルールベースで参加条件を自動チェック |
| **判定ルール** | ① 全省庁統一資格の記載あり ② 等級が自身の等級以上 ③ 営業品目が合致 ④ 競争参加地域に該当 ⑤ 許認可要件を満たす |
| **出力** | 判定結果（参加可能/参加不可＋理由）＋根拠箇所 |

#### F-004: チェックリスト＋応札準備テンプレ自動生成

| 項目 | 内容 |
|------|------|
| **入力** | 案件カード（F-002の出力） |
| **処理** | ① 提出物リストの抽出 ② 各提出物のテンプレート・書式ルールの適用 ③ 期限の逆算スケジュール生成 |
| **出力** | チェックリスト（提出物名・期限・ステータス・テンプレートリンク） |
| **知識ベース** | ホットプラスのノウハウ: 入札書の記入ルール、封筒の二重構造ルール、下見積もり書のルール等 |

#### F-005: 落札実績データの取り込み＋価格分析

| 項目 | 内容 |
|------|------|
| **入力** | 調達ポータル落札実績オープンデータ（CSV/JSON） |
| **処理** | ① 定期取得（全件/差分） ② 正規化・DB格納 ③ 対象領域の案件について落札金額・参加社数・落札業者を集計 |
| **出力** | 価格分析レポート（相場帯、参加社数分布、勝率推定） |
| **更新頻度** | 差分データの定期取得（週次〜日次） |

## 5. 非機能要件

### 5.1 データ取得・信頼性

| 要件 | 内容 |
|------|------|
| 原本保存 | 取得した公告HTML/PDF/CSV の原本を必ず保存（監査・再現性） |
| 根拠保持 | AI抽出結果は必ず根拠箇所（URL/ページ/見出し/引用）を保持 |
| 差分検知 | 公開サイトの更新を検知し、差分のみ取得 |
| 重複排除 | 同一案件の重複登録を防止 |
| 取得失敗対応 | リトライ（最大3回）、失敗時アラート、手動取得のフォールバック |
| 構造変更対応 | 取得先サイトの構造変更を検知し、アラート発報 |

### 5.2 AI読解・安全性

| 要件 | 内容 |
|------|------|
| 断定しない | AI要約は「〜と記載されています」「〜の可能性があります」等の表現を使用 |
| 原文確認導線 | 抽出結果から原文（PDF/HTML）へのリンクを必ず提供 |
| ハルシネーション対策 | 根拠が見つからない情報は「根拠なし」と明示。推測と事実を区別 |
| 人間確認前提 | AI判定を最終判断としない。チェックリストに「人間確認」ステップを含める |

### 5.3 セキュリティ

| 要件 | 内容 |
|------|------|
| 認証 | Phase1は単一ユーザー。最小限の認証で開始。Phase2でマルチテナント対応 |
| データ保護 | 入札情報（公開情報）は機密度低。ただしユーザーの資格情報・入札戦略は保護対象 |
| API キー管理 | LLM API キー等は環境変数で管理。ハードコード禁止 |
| ログ | 全API呼び出し・データ取得・AI読解のログを保持 |

### 5.4 パフォーマンス

| 要件 | 内容 |
|------|------|
| 案件一覧表示 | 2秒以内（キャッシュ済みデータ） |
| AI読解（1案件） | 30秒以内（LLM API呼び出し含む） |
| 日次バッチ（案件収集） | 30分以内 |
| 同時ユーザー | Phase1: 1人。Phase2: 100人を目標 |

## 6. Phase分割とスコープ

### 6.1 Phase 一覧

| Phase | 名称 | 目的 | 期間目安 | 対象ユーザー |
|-------|------|------|---------|-------------|
| 0 | データ検証 | 案件件数・AI読解精度・データ取得可否の検証 | 2〜4週間 | 自分のみ |
| 1 | 自社利用MVP | 実際に入札に参加し、落札する | 3〜6ヶ月 | 自分のみ |
| 2 | SaaS化 | 同じ課題を持つ中小事業者にSaaS展開 | Phase1完了後 | 入札初心者 |
| 3 | コンサルJV | コンサル会社との共同事業 | Phase2安定後 | コンサル会社 |

### 6.2 Phase0: データ検証

**目的**: MVPの実現可能性を技術的に検証する

| # | 検証項目 | 方法 | 成功基準 |
|---|---------|------|---------|
| 1 | 対象3領域の案件件数 | データウェアネット＋調達ポータルでの件数カウント | 月20件以上 |
| 2 | 調達ポータルODの取り込み | CSV/JSONの定期取得テスト | 差分取得が安定動作 |
| 3 | AI読解精度 | 実際の仕様書PDFをLLMに読ませて精度確認 | 要点抽出の精度80%以上 |

**Go/No-Go 判断**:
- Go: 3項目すべて成功基準クリア → Phase1 開発開始
- No-Go: 1項目でも大幅未達 → 対象領域の変更 or アプローチの見直し

### 6.3 Phase1: 自社利用MVP

**スコープ**: P0機能（F-001〜F-005）の開発・運用

**Phase1 の特徴**:
- ユーザーは自分1人。マルチテナント対応は不要
- UIはシンプルでよい（自分が使えれば十分）
- 品質基準は「自分の入札事業が回るか」で判定
- AI読解の精度は人間確認前提で段階的に向上

**Phase1 完了条件**:
- 月10件以上の入札参加
- 月2件以上の落札
- 案件探索時間が1日10分以内

### 6.4 Phase2: SaaS化

**スコープ**: P1機能追加 + マルチテナント対応 + 課金

**Phase2 の前提条件**:
- Phase1で月2件以上の落札を達成
- SaaS化の需要をインタビュー・アンケートで検証

**プラン設計**:

> 詳細: VALUE_PROPOSITION.md §3.5

| プラン | 対象 | 主要機能 | 想定月額 | 根拠 |
|-------|------|---------|---------|------|
| Starter | 入札初心者 | 案件カード＋根拠提示＋チェックリスト＋参加可否判定 | ¥9,800〜14,800 | 時短価値(月40h×¥2,000=¥80,000)の12〜18% |
| Pro（本命） | 入札事業者 | + 失注学習ループ＋テンプレ生成＋外注/協力会社連携 | ¥29,800〜49,800 | 追加落札額(¥300,000〜)の10〜16% |
| Biz（B向け） | コンサル/企業 | + 会員テンプレ＋横断ダッシュボード＋監査/権限 | ¥98,000〜 | 人件費削減(3名×40h)の40% |

**価格戦略**: 低価格で勝負しない。費用対効果（時短×リスク低減×勝率）ベース。
Phase1のMVPで成果指標（工数削減率・不備率・落札率）を計測してから確定。

> ※ 上記は計測前の仮説であり、Phase1 の成果指標で検証する。

### 6.5 Phase3: コンサルJV

**スコープ**: P2機能 + マルチアカウント + 横断ダッシュボード

**按分**: 4:6（JV先:弊社）

## 8. リスクと前提

### 8.1 主要リスク

| # | リスク | 影響度 | 発生可能性 | 対策 |
|---|-------|-------|-----------|------|
| 1 | データ取得の網羅性・正確性 | 高 | 高 | 差分検知・重複排除・原文保存・失敗時リトライ。対象3領域に集中 |
| 2 | AI読解の精度（仕様書の誤読→無効入札） | 高 | 中 | 根拠保持＋断定しない設計。人間の最終確認を前提 |
| 3 | 対象領域の案件数が不足 | 中 | 中 | Phase0で件数検証。不足時は領域拡大 |
| 4 | 取得先サイトの構造変更 | 中 | 高 | 構造変更検知＋アラート。原本保存で再パース可能な設計 |
| 5 | 1人開発の限界（開発速度・保守） | 中 | 高 | MVP機能を5つに絞る。Phase1は自分だけが使う前提で品質基準を現実的に設定 |
| 6 | 電子入札の認証・規約リスク | 低 | — | 提出行為は自動化しない。「準備の自動化」に徹する |

### 8.2 前提条件

| # | 前提 | 検証方法 | 検証時期 |
|---|------|---------|---------|
| 1 | 調達ポータルの落札実績OD が引き続き公開される | 公式サイトを定期確認 | 継続的 |
| 2 | LLM API（Claude等）が仕様書読解に十分な精度を持つ | Phase0 PoC | Phase0 |
| 3 | 下請けNW（義父:軽運送、知人:内装・電気）が案件実行に協力可能 | 直接確認 | Phase1 |
| 4 | ホットプラスのノウハウが入札フロー全体をカバーしている | KNOWLEDGE_DIGEST.md で確認済み | 確認済み |
