### 設計方針: 判定「補助」（Human-in-the-loop）

> **F-003は自動判定ではなく判定補助である。**
> - AIが最終判断を下さない。判定結果はあくまで「推奨」
> - `uncertain`（判定不能）を設計上の第一級市民として扱う
> - 曖昧な条件・欠損データは無理に判定せず `uncertain` に落とす
> - 人間が確認・上書きするまで判定は暫定

## 2. ユーザーストーリー

### メインストーリー

> **ペルソナ1（金子）として、**
> F-002が読み解いた案件の参加条件を自分の資格と突き合わせて、
> 「参加できる」「できない」「確認が必要」をひと目で判断できれば、
> 対象外の案件に時間をかけずに済み、参加可能な案件に集中できる。

### 具体シナリオ

**シナリオ1: 明確に参加可能 — eligible**
1. 案件「○○省 軽運送業務委託」の案件カードを表示
2. 参加可否判定が自動で実行され、結果が表示される:
   - 判定: ✅ **eligible（参加可能）**
   - 必須条件チェック:
     - 全省庁統一資格: ✅ あり（公告p.1に記載）
     - 等級: ✅ D等級以上（自社D等級で適合）
     - 営業品目: ✅ 「その他」で参加可能
     - 地域: ✅ 関東甲信越（自社: 関東甲信越）
     - 許認可: ✅ 要件なし
   - 推奨条件: 特になし
3. → 安心して応札準備に進める

**シナリオ2: 明確に参加不可 — ineligible**
1. 案件「△△庁 ITシステム保守」の案件カードを表示
2. 判定結果:
   - 判定: ❌ **ineligible（参加不可）**
   - 不可理由:
     - 等級: ❌ **A等級が必要**（自社D等級で不適合）— 根拠: 公告p.1「A等級」
   - hard_fail_reasons: `["grade_mismatch: required=A, mine=D"]`
3. → 即座に見送り判断ができる

**シナリオ3: 判定不能 — uncertain**
1. 案件「□□省 施設管理業務」の案件カードを表示
2. 判定結果:
   - 判定: ⚠️ **uncertain（要確認）**
   - 確認が必要な理由:
     - 許認可: ⚠️ **「建築物環境衛生管理技術者の配置」が必要**（自社保有状況: 未登録）
     - 実績: ⚠️ **「同種業務の履行実績」が求められている**（自社実績: 未登録）
   - soft_gaps: `["license_unknown: 建築物環境衛生管理技術者", "experience_unknown: 同種業務実績"]`
3. → 許認可と実績を確認した上で判断する

**シナリオ4: company_profile 未登録 — 自動 uncertain**
1. company_profile が未設定の状態で判定を実行
2. 判定結果:
   - 判定: ⚠️ **uncertain（要確認）**
   - 理由: 「会社プロフィールが未登録のため判定できません」
3. → プロフィール設定画面への導線を表示

## 3. 機能要件

### 3-A. 入力

#### 3-A-1. F-002からの連携データ（案件カード eligibility）

| データ | 型 | 説明 | 用途 |
|-------|-----|------|------|
| case_id | UUID | 案件ID | 結果の紐付け |
| eligibility | JSONB | 参加条件の構造化データ | 判定ルールの入力 |
| eligibility.unified_qualification | BOOLEAN | 全省庁統一資格の記載有無 | Hard条件①チェック |
| eligibility.grade | VARCHAR | 等級（A/B/C/D） | Hard条件②チェック |
| eligibility.business_category | VARCHAR | 営業品目 | Hard条件③チェック |
| eligibility.region | VARCHAR | 競争参加地域 | Hard条件④チェック |
| eligibility.additional_requirements | JSONB | その他参加要件（許認可・実績・ISO等） | Hard条件⑤ + Soft条件チェック |
| evidence（eligibility部分） | JSONB | 参加条件の根拠情報 | 判定理由の根拠表示 |
| confidence_score | DECIMAL | F-002読解全体の信頼度 | 低信頼度時の uncertain 判定 |
| assertion_counts | JSONB | fact/inferred/caution集計 | 信頼度の補助判定 |

#### 3-A-2. 会社プロフィール（company_profile）

> **参加可否は案件情報だけでは判定できない。自社の資格・体制と突き合わせる必要がある。**

| フィールド | 型 | 説明 | 必須 | Phase1の値（金子） |
|-----------|-----|------|-----|-------------------|
| unified_qualification | BOOLEAN | 全省庁統一資格の保有 | ○ | true |
| grade | VARCHAR | 等級（A/B/C/D） | ○ | D |
| business_categories | JSONB | 保有営業品目のリスト | ○ | ["物品の販売", "役務の提供その他"] |
| regions | JSONB | 競争参加地域のリスト | ○ | ["関東・甲信越"] |
| licenses | JSONB | 保有許認可のリスト | △ | []（全省庁統一資格のみ。その他の許認可なし） |
| certifications | JSONB | ISO/Pマーク等の認証 | △ | [] |
| experience | JSONB | 過去の官公庁実績 | △ | []（入札経験なし） |
| subcontractors | JSONB | 協力会社の体制・資格 | △ | [{"name": "クローバー運輸", "license": "運送業"}, {"name": "電気工事会社", "license": "電気工事業"}, {"name": "内装関係", "license": "内装業"}] |
| updated_at | TIMESTAMP | 最終更新日 | ○ | — |

**欠損時の挙動:**
- 必須項目（上4つ）が未設定 → **全案件 uncertain**（判定不能）
- 任意項目（下4つ）が未設定 → 該当条件を **uncertain** 扱い（条件に一致しなかった場合、不可ではなく未確認）

### 3-B. 処理

#### 3-B-1. 判定パイプライン

```
═══════════════════════════════════════════════
  Phase 1: 前提チェック
═══════════════════════════════════════════════

[案件カード(case_cards) + 会社プロフィール(company_profile)]
       │
       ▼
  ① company_profile の存在チェック
     → 必須4項目が全て設定されているか
     → 未設定 → 即座に verdict = "uncertain"
       + reason = "company_profile_incomplete"
       │
       ▼
  ② F-002 読解結果の信頼度チェック
     → confidence_score < 0.6 の場合
       → verdict = "uncertain"
       + reason = "low_extraction_confidence"
     → eligibility が NULL or 空の場合
       → verdict = "uncertain"
       + reason = "no_eligibility_data"

═══════════════════════════════════════════════
  Phase 2: Hard条件チェック（必須条件）
═══════════════════════════════════════════════

  ③ 5つの必須条件を順次チェック（1つでも不可 → ineligible）

  [Hard-1] 全省庁統一資格
     → 案件: unified_qualification = true
     → 自社: unified_qualification = true
     → 不一致: hard_fail（「全省庁統一資格が必要ですが未保有です」）
     → 案件側が null/inferred: uncertain（「統一資格の要否を確認してください」）

  [Hard-2] 等級
     → 案件: grade = "D" (例)
     → 自社: grade = "D"
     → 判定: 自社等級 ≦ 要求等級（A<B<C<D の順で D が最も条件緩い）
     → 不一致: hard_fail（「A等級が必要ですが自社はD等級です」）
     → 案件側が null: uncertain

  [Hard-3] 営業品目
     → 案件: business_category = "物品の販売"
     → 自社: business_categories に含まれるか
     → 「その他」はほぼ全てに該当（KNOWLEDGE_DIGEST参照）
     → 不一致: hard_fail
     → 案件側が null: uncertain

  [Hard-4] 競争参加地域
     → 案件: region = "関東・甲信越"
     → 自社: regions に含まれるか
     → 不一致: hard_fail
     → 案件側が null: uncertain

  [Hard-5] 許認可（additional_requirements の許認可部分）
     → 案件: additional_requirements 内の許認可項目を抽出
     → 自社: licenses に含まれるか
     → 不一致: hard_fail
     → 自社の licenses が空 & 案件に許認可要件あり: uncertain（欠損による）
     → 案件側に許認可要件なし: pass

  ※ Hard条件で 1つでも hard_fail → verdict = "ineligible"
  ※ Hard条件で hard_fail なし & 1つ以上 uncertain → Phase 3 へ（暫定uncertain）

═══════════════════════════════════════════════
  Phase 3: Soft条件チェック（推奨・加点条件）
═══════════════════════════════════════════════

  ④ additional_requirements の中から Soft条件を抽出・評価

  [Soft-1] 実績要件
     → 案件: 「同種業務の履行実績」等の記載
     → 自社: experience に該当実績があるか
     → なし: soft_gap（「同種業務実績が求められています」）
     → 判定: 実績要件が「必須」か「望ましい」かで重みを分ける
       - 「必須」記載 → soft_gap (severity=high) → uncertain に寄与
       - 「望ましい」記載 → soft_gap (severity=low) → 情報提供のみ

  [Soft-2] ISO/Pマーク
     → 案件: additional_requirements に ISO/Pマーク記載
     → 自社: certifications に含まれるか
     → なし: soft_gap
     → 注意: 実質必須の場合もある（assertion_type で判断）

  [Soft-3] 所在地要件
     → 案件: 「○○県内に事業所を有する者」等
     → 自社: company_profile に所在地情報があれば照合
     → 情報なし: soft_gap (severity=medium)

  [Soft-4] 体制・人員要件
     → 案件: 「技術者○名以上」「責任者の配置」等
     → 自社: subcontractors の体制で充足できるか
     → 情報なし: soft_gap (severity=medium)

═══════════════════════════════════════════════
  Phase 4: 最終判定
═══════════════════════════════════════════════

  ⑤ 判定ロジック

  IF hard_fail_reasons が 1件以上
    → verdict = "ineligible"
    → confidence = min(各Hard条件のevidence confidence)

  ELSE IF hard条件に uncertain が 1件以上
       OR soft_gap に severity=high が 1件以上
    → verdict = "uncertain"
    → confidence = 計算（後述）

  ELSE IF soft_gap が 0件
    → verdict = "eligible"
    → confidence = min(各条件のevidence confidence)

  ELSE (soft_gap が severity=low/medium のみ)
    → verdict = "eligible"  ← 参加可能だが注意事項あり
    → soft_gaps をレポートに含める
    → confidence = min(各条件のevidence confidence) × 0.9

  ⑥ DB格納: eligibility_results テーブル
```

#### 3-B-2. confidence の算出

```
判定 confidence = min(
  F-002 confidence_score,           // 読解の信頼度
  min(各Hard条件の evidence confidence), // 根拠の確かさ
  company_profile 完全度スコア       // 0.0〜1.0（必須4/4=1.0、任意欠損で減点）
)
```

- company_profile 完全度: 必須4項目=各0.25、任意4項目=各0.0625（合計0.25分のボーナス）
  - 必須全項目あり: 1.0
  - 任意も全項目あり: 1.0（上限は1.0）
  - 必須1項目欠損: 0.75 → uncertain 確定

### 3-D. ビジネスルール

#### 判定の基本原則

| # | ルール | 詳細 |
|---|-------|------|
| 1 | **uncertain は第一級市民** | 曖昧な条件・欠損データは無理に判定しない。`uncertain` に落とし、理由を明示する |
| 2 | **Hard/Soft の分離** | 必須条件（Hard）の不合格は即座に `ineligible`。推奨条件（Soft）は `soft_gaps` として報告し、判定には重み付けで反映 |
| 3 | **人間上書き優先** | `human_override` が設定された場合、自動判定より優先。上書き理由を記録 |
| 4 | **根拠追跡** | 全判定項目に F-002 evidence へのポインタ（evidence_ref）を保持。判定理由を原文まで追える |
| 5 | **assertion_type 連動** | F-002で `inferred` / `caution` の条件は、判定で `uncertain` に寄与する重みを上げる |
| 6 | **再判定** | F-002が再読解された場合、または company_profile が更新された場合に自動再判定。旧判定は履歴として保持 |
| 7 | **company_profile スナップショット** | 判定時点の company_profile を保存し、後からの変更で判定根拠が不明にならないようにする |

#### Hard条件の判定ルール詳細

| # | 条件 | 判定ロジック | 不合格時の扱い | 根拠 |
|---|------|------------|-------------|------|
| Hard-1 | 全省庁統一資格 | 案件で required & 自社で保有 | hard_fail | KNOWLEDGE_DIGEST: 参加条件チェック①  |
| Hard-2 | 等級 | 自社等級 ≦ 要求等級（A<B<C<D） | hard_fail | KNOWLEDGE_DIGEST: 参加条件チェック③ |
| Hard-3 | 営業品目 | 案件品目 ∈ 自社品目リスト。「その他」は物品の販売・役務の提供いずれでもワイルドカード | hard_fail | KNOWLEDGE_DIGEST: 参加条件チェック② |
| Hard-4 | 地域 | 案件地域 ∈ 自社地域リスト | hard_fail | KNOWLEDGE_DIGEST: 参加条件チェック④ |
| Hard-5 | 許認可 | 案件の許認可要件 ⊆ 自社許認可。自社側欠損時は uncertain | hard_fail or uncertain | KNOWLEDGE_DIGEST: その他の参加資格 |

> **等級の順序**: A（最上位）> B > C > D（最下位）。D等級はほぼ全ての案件に参加可能だが、A等級が要求される案件には不可。

#### Soft条件の severity 判定

| severity | 基準 | verdict への影響 |
|----------|------|----------------|
| high | 「必須」「〜を有する者」等の強い表現。assertion_type = fact | uncertain に寄与 |
| medium | 「望ましい」「〜が可能な者」等の推奨表現。assertion_type = fact/inferred | 情報提供（verdict に軽微な影響） |
| low | 加点要素・優遇条件。assertion_type = any | 情報提供のみ（verdict に影響なし） |

> - 判定ロジックの例外（予期しない値）: verdict = "uncertain" + エラーログ + アラート
> - DB接続エラー: リトライ3回 → 失敗ログ

## 4. 非機能要件

| カテゴリ | 要件 | 目標値 |
|---------|------|-------|
| 判定速度（1案件） | Hard 5条件 + Soft 4条件のチェック | 1秒以内（ルールベースのためLLM不使用） |
| バッチ判定 | F-002読解完了後の自動トリガー | 全対象案件を5分以内に判定完了 |
| 判定精度 | Hard条件の判定正確性 | 99%以上（ルールベース＋根拠あり = 高精度） |
| uncertain 率 | 全案件中の uncertain の割合 | Phase1初期: 30%以下を目標（company_profile充実で低下） |
| 同時処理 | 判定の並列数 | Phase1: 逐次処理 |
| 監視 | 判定結果の分布 | eligible/ineligible/uncertain の比率をログ |
| 履歴保持 | 再判定時の旧結果 | 旧結果は soft delete（deleted_at）で保持 |

### 6-1. Phase0（M6）での検証項目

| # | 検証項目 | 内容 | 成功基準 |
|---|---------|------|---------|
| V-301 | Hard条件の判定正確性 | 実際の公告10件で人間判定と比較 | 一致率 95%以上 |

| V-304 | Soft条件の severity 判定精度 | additional_requirements の分類精度 | high/medium/low の誤分類 10%以下 |
| V-305 | 判定速度 | 10案件の一括判定 | 10秒以内 |

### 6-2. 技術選択の方針

| 項目 | 方針 | 備考 |
|------|------|------|
| DB | **PostgreSQL（全機能で共通）** | eligibility_results + company_profile テーブル |
| 判定エンジン | **ルールベース（Python）** | LLM不使用。条件分岐の純粋なロジック |
| Soft条件の severity 判定 | assertion_type + キーワードマッチ | 「必須」「有する者」→ high、「望ましい」「可能な」→ medium/low |
| company_profile 管理 | Phase1: 設定ファイル or DB直接。Phase2: UI から編集 | |
| テスト | pytest + テーブル駆動テスト | 各条件の組み合わせを網羅 |

### 6-3. 開発順序（F-003内部）

```
基盤:
  1. company_profile テーブル定義 + 初期データ投入（金子の情報）
  2. eligibility_results テーブル定義

判定エンジン:
  3. Phase 1: 前提チェック（company_profile存在 + confidence閾値）
  4. Phase 2: Hard条件チェック（5条件の判定ロジック）
  5. Phase 3: Soft条件チェック（4条件 + severity判定）
  6. Phase 4: 最終判定（verdict + confidence算出）
  7. 人間上書きロジック（human_override の設定・反映）

連携:
  8. F-002 → F-003 トリガー（case_cards 作成/更新時に自動判定）
  9. F-003 → F-001 反映（案件カードへの判定結果表示）
  10. F-003 → F-004 トリガー（eligible 時のチェックリスト生成起動）
  11. 再判定ロジック（F-002再読解 or company_profile更新時）
```

### 6-4. 将来拡張

- **マルチテナント対応（Phase2）**: company_profile を組織単位で管理。ユーザーごとに異なる資格・体制
- **学習ループ（F-009連携）**: 判定結果と実際の参加可否の突合 → 判定ルールの精度向上
- **総合評価方式対応**: 技術点の加点要素もSoft条件として評価
- **AI補助判定**: additional_requirements の曖昧な記載をLLMで解釈し、severity を自動判定（Phase2以降）
