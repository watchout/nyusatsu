## 2. ユーザーストーリー

### メインストーリー

> **ペルソナ1（金子）として、**
> F-003で「参加可能」と判定された案件について、
> 「何を・いつまでに・どう準備するか」のチェックリストが自動で生成されれば、
> 入札書の書き方や封筒のルールを調べる手間がなくなり、
> 初めての入札でも抜け漏れなく準備が進められる。

### 具体シナリオ

**シナリオ1: チェックリスト自動生成 — 何を準備するか一目でわかる**
1. F-003で「eligible」と判定された案件を開く
2. チェックリストが自動生成されている:
   - **入札参加フェーズ**:
     - [ ] 入札書の記入（発注機関書式）— 期限: 2026-03-15
     - [ ] 委任状の準備 — 期限: 2026-03-15
     - [ ] 全省庁統一資格の写し — 期限: 2026-03-15
     - [ ] 封筒の準備（内封筒＋外封筒の二重構造）— 期限: 2026-03-15
     - [ ] ⚠️ 下見積もり書の提出 — **期限: 2026-03-05**（入札書より先！）
   - **履行フェーズ**（落札後）:
     - [ ] 作業者名簿の提出
     - [ ] 業務計画書の作成
     - [ ] 保険証の写し
3. → 何を準備すればいいかが一瞬でわかる

**シナリオ2: 期限逆算スケジュール — いつまでにやるか**
1. チェックリストの「スケジュール」タブを確認
2. deadline_at（入札書提出期限: 2026-03-15）から4段階逆算:
   - 2026-02-26: 下見積もり書 作成開始（quote_deadline -5BD）
   - 2026-03-03: 下見積もり書 社内レビュー（quote_deadline -2BD）🔴
   - 2026-03-05: **下見積もり書 提出期限**（入札書より先！）🔴
   - 2026-03-10: 入札書・封筒 作成開始（deadline_at -5BD）
   - 2026-03-13: 社内レビュー（deadline_at -2BD）🔴
   - 2026-03-14: 最終確定・封緘（deadline_at -1BD）🔴
   - 2026-03-15: **入札書提出期限**（deadline_at 当日）🔴
   - 2026-03-20: 開札日
   - 2026-04-30: 履行期限
3. → 逆算で「いつ着手すべきか」がわかる。🔴マークは critical ステップ

**シナリオ3: テンプレート参照 — どう書くか**
1. チェックリストの「入札書の記入」をクリック
2. ナレッジに基づくガイドが表示される:
   - 入札書: 発注機関のひな形を使用。機関指定書式以外は無効の場合あり
   - 封筒: 内封筒（長3）＋外封筒の二重構造。表面に入札番号・案件名を記入
   - 提出方法: 書留 or レターパック（事前許可要）
3. → 初めてでも書き方がわかる

**シナリオ4: 進捗管理 — どこまで終わったか**
1. 準備が完了した項目にチェックを入れる: ✅ 入札書の記入
2. 進捗率が更新される: 2/5 完了（40%）
3. 期限が迫っている未完了項目はハイライト表示
4. → 準備の抜け漏れを防止

**シナリオ5: 物品案件 — 下見積もり書の注意喚起**
1. 物品案件で has_quote_requirement = true の場合
2. チェックリストの最上部に警告:
   - ⚠️ **下見積もり書の提出が必要です**（期限: 2026-03-05）
   - ⚠️ **下見積もり書の金額は入札参加額より高い金額を記入すること**
3. → 致命的なミスを防止

## 3. 機能要件

### 3-A. 入力

#### 3-A-1. F-002からの連携データ（案件カード）

| データ | 型 | 説明 | 用途 |
|-------|-----|------|------|
| case_id | UUID | 案件ID | 結果の紐付け |
| schedule | JSONB | スケジュール情報 | 期限逆算スケジュール生成 |
| submission_items | JSONB | 提出物リスト（bid_time + performance_time） | **チェックリスト項目の直接入力** |
| business_content | JSONB | 業務内容 | テンプレート選択（物品/役務で分岐） |
| risk_factors | JSONB | リスク要因 | 警告マーカー表示 |
| evidence | JSONB | 全根拠情報 | 各チェック項目の根拠として表示 |
| deadline_at | TIMESTAMP | 最重要期限（= submission_deadline） | 逆算スケジュールの基準日 |
| assertion_counts | JSONB | fact/inferred/caution集計 | チェックリストの信頼度表示 |

#### 3-A-2. F-003からの連携データ（参加可否判定）

| データ | 型 | 説明 | 用途 |
|-------|-----|------|------|
| verdict | VARCHAR | 判定結果（eligible/ineligible/uncertain） | **生成トリガー**（eligible時のみ自動生成） |
| soft_gaps | JSONB | 推奨条件の不足リスト | 追加確認事項として挿入 |

#### 3-A-3. ナレッジベース（テンプレート・ルール）

| ナレッジ | 用途 | ソース |
|---------|------|-------|
| 入札書記入ルール | 記入ガイドの生成 | KNOWLEDGE_DIGEST.md STEP4 |
| 封筒ルール | 封筒準備ガイドの生成 | KNOWLEDGE_DIGEST.md STEP4（二重構造） |
| 提出方法ルール | 書留/レターパックの選択ガイド | KNOWLEDGE_DIGEST.md STEP4 |
| 下見積もり書ルール | 金額記入ルールの注意喚起 | KNOWLEDGE_DIGEST.md STEP3 |
| 履行時の提出物 | 作業者名簿等のガイド | KNOWLEDGE_DIGEST.md STEP6 |

### 3-B. 処理

#### 3-B-1. チェックリスト生成パイプライン

```
═══════════════════════════════════════════════
  Phase 1: トリガー判定
═══════════════════════════════════════════════

[F-003 verdict = "eligible" を受信]
       │
       ▼
  ① トリガー判定
     → verdict = "eligible" → Phase 2 へ
     → verdict = "uncertain" + human_override = "eligible" → Phase 2 へ
     → verdict = "ineligible" or "uncertain" → 生成しない

═══════════════════════════════════════════════
  Phase 2: 提出物 → チェックリスト変換
═══════════════════════════════════════════════

[F-002 submission_items を取得]
       │
       ▼
  ② 入札参加フェーズの項目変換
     → submission_items.bid_time_items を展開
     → 各提出物を checklist_item に変換:
       - name: 提出物名
       - phase: "bid_time"
       - deadline: submission_items 内の個別期限、なければ deadline_at
       - template_type: 発注機関書式 / 汎用 / なし
       - notes: assertion_type に基づく注意書き
       - evidence_ref: F-002 evidence へのポインタ
       - status: "pending"
       │
       ▼
  ③ 履行フェーズの項目変換
     → submission_items.performance_time_items を展開
     → 各提出物を checklist_item に変換:
       - phase: "performance_time"
       - deadline: NULL（落札後に確定）
       │
       ▼
  ④ 固定項目の追加（ナレッジベースから）
     → 入札書が提出物に含まれている場合:
       - 「封筒の準備（内封筒＋外封筒の二重構造）」を追加
       - 「提出方法の確認（書留 or レターパック）」を追加
     → has_quote_requirement = true の場合:
       - 「下見積もり書」を最優先項目として追加
       - ⚠️ 「金額は入札参加額より高い金額を記入」の警告を付与
       │
       ▼
  ④-b 確認タスクの自動挿入（uncertain案件のみ）
     → F-003 verdict = "uncertain" かつ human_override = "eligible" の場合:
       - チェックリスト最上部に「確認タスク」を挿入（下見積もり書より上）
       - 確認タスクは source = "auto_confirm" で区別

     [確認タスク生成ロジック]
     | F-003の不確実性の理由 | 生成されるタスク |
     |---------------------|---------------|
     | company_profile 欠損（Phase 1で insufficient_data） | 「❓ 資格区分/等級/品目/地域を確認して登録」 |
     | Hard条件が unknown（check_details に unknown あり） | 「❓ 仕様書の該当箇所（根拠）を確認して判定を上書き」 |
     | Soft条件に warning あり | 「❓ [条件名]の充足を確認」 |

     [確認タスクの checklist_item 構造]
     ```json
     {
       "item_id": "confirm_001",
       "name": "❓ 資格区分/等級/品目/地域を確認して登録",
       "phase": "bid_time",
       "deadline": null,
       "recommended_start": null,
       "template_type": null,
       "template_url": null,
       "template_guide": "F-003の判定結果を確認し、不足情報を補完してください",
       "status": "pending",
       "source": "auto_confirm",
       "warnings": ["⚠️ uncertain判定のため、手動確認が必要です"],
       "evidence_ref": null
     }
     ```

     → eligible（確定）案件の場合は確認タスクなし（通常生成）

═══════════════════════════════════════════════
  Phase 3: 期限逆算スケジュール生成
═══════════════════════════════════════════════

  ⑤ deadline_at を基準に逆算
     → 各提出物の推奨着手日を算出:

     [逆算ルール — 4段階デフォルト（営業日ベース）]

     ■ 入札書提出の逆算（deadline_at 基準）:
     | ステップ | 基準日 | 日数 | is_critical | 説明 |
     |---------|--------|------|-------------|------|
     | 提出物作成開始 | deadline_at - 5営業日 | -5BD | false | 入札書記入・封筒準備等の着手推奨日 |
     | 社内レビュー | deadline_at - 2営業日 | -2BD | true | 記入内容・金額の最終チェック |
     | 最終確定 | deadline_at - 1営業日 | -1BD | true | 封緘・発送準備 |
     | 提出当日タスク | deadline_at 当日 | 0BD | true | 提出（書留/レターパック/持参） |

     ■ 下見積もり書の逆算（quote_deadline 基準）:
     → has_quote_requirement = true の場合、quote_deadline を基準に同じ4段階を生成
     | ステップ | 基準日 | 日数 | is_critical |
     |---------|--------|------|-------------|
     | 下見積もり作成開始 | quote_deadline - 5営業日 | -5BD | false |
     | 社内レビュー | quote_deadline - 2営業日 | -2BD | true |
     | 最終確定 | quote_deadline - 1営業日 | -1BD | true |
     | 下見積もり提出 | quote_deadline 当日 | 0BD | true |

     ■ イベント日（逆算なし・参考表示）:
     - 入札書提出期限: deadline_at（基準日）
     - 開札日: schedule.opening_date
     - 履行開始: schedule.performance_deadline（参考表示）

     ■ 上書きルール:
     → 将来的に発注機関別・案件カテゴリ別のデフォルト日数で上書き可能
     → Phase1 は上記の固定値で運用

     [営業日計算]
     - 土日祝を除く
     - Phase1: 日本の祝日テーブルを参照（jpholiday ライブラリ）

═══════════════════════════════════════════════
  Phase 4: 警告・補足情報の付与
═══════════════════════════════════════════════

  ⑥ risk_factors の反映
     → 各リスクを対応するチェック項目の警告として表示
     → 例: 「下見積もり期限迫り」→ 下見積もり書項目にハイライト
       │
       ▼
  ⑦ soft_gaps の反映（F-003から）
     → 追加確認事項としてチェックリストの末尾に挿入
     → 例: 「同種業務実績が求められています」→ 「実績書類の準備を検討」
       │
       ▼
  ⑧ assertion_counts による信頼度表示
     → inferred/caution が多い場合:
       - チェックリスト冒頭に「⚠️ AI抽出の一部に推定が含まれます。原文を確認してください」
       │
       ▼
  ⑨ DB格納: checklists テーブル
```

#### 3-B-2. テンプレートマッピング

| テンプレート種別 | 適用条件 | Phase1での提供 |
|----------------|---------|--------------|
| 発注機関指定書式 | 公告・仕様書に書式リンクあり | URL参照（自動DLは Phase2） |
| 汎用入札書テンプレート | 発注機関書式が不明 | ✅ Phase0で作成: 記入ガイド付き汎用版（STEP4ナレッジ参照）。発注機関指定がない場合の保険 |
| 封筒テンプレート | 入札書の提出がある場合 | ✅ Phase0で作成: 表/内封筒の記入例＋注意事項。二重構造ルール（内封筒:長3 + 外封筒）対応 |
| 下見積もり書テンプレート | has_quote_requirement = true | ✅ Phase0で作成: 汎用版＋金額ルール警告（入札参加額より高い金額を記入）付き（STEP3ナレッジ参照） |
| 委任状テンプレート | 代理提出の場合 | 汎用テンプレート |

> **テンプレート優先度**: 発注機関指定書式 > 汎用テンプレート

### 3-D. ビジネスルール

#### チェックリスト生成ルール

| # | ルール | 詳細 |
|---|-------|------|
| 1 | **フェーズ分離** | 入札参加フェーズ（bid_time）と履行フェーズ（performance_time）を明確に分離。UIでタブまたはセクション分け |
| 2 | **自動生成** | F-002の submission_items から自動生成。手動でチェックリストを1から作成する必要はない |
| 3 | **トリガー** | F-003 verdict = "eligible"（または human_override = "eligible"）で自動生成。uncertain/ineligible では生成しない |
| 4 | **1案件1チェックリスト** | 1つの cases レコードに対して1つの checklists レコード。再生成時は UPSERT（手動チェック状態は保持） |
| 5 | **手動追加可** | ユーザーがチェック項目を手動追加可能。手動追加項目は `source: "manual"` で区別 |
| 6 | **進捗計算** | `progress.rate = done / total`。全項目完了で status = "completed" |

#### 下見積もり書の特別ルール（KNOWLEDGE_DIGEST STEP3 参照）

### 6-1. Phase0（M6）での検証項目

### 6-2. 技術選択の方針

| 項目 | 方針 | 備考 |
|------|------|------|
| DB | **PostgreSQL（全機能で共通）** | checklists テーブル |
| 生成エンジン | **ルールベース（Python）** | LLM不使用。submission_items の変換 + 固定ルール |
| 営業日計算 | Python `jpholiday` ライブラリ | 日本の祝日対応 |
| テンプレート管理 | Phase1: Markdown/YAML ファイルで管理。Phase2: DB管理 + UI |
| 進捗管理 | DB の JSONB 更新（checklist_items 内の status フィールド） | |

### 6-3. 開発順序（F-004内部）

```
基盤:
  1. checklists テーブル定義
  2. checklist_item の JSONB スキーマ定義（pydantic）
  3. schedule_item の JSONB スキーマ定義（pydantic）

生成エンジン:
  4. submission_items → checklist_items 変換ロジック
  5. 固定項目の追加ロジック（封筒・提出方法・下見積もり書）
  6. 期限逆算スケジュール生成（営業日計算含む）
  7. 警告・補足情報の付与（risk_factors + soft_gaps + assertion_counts）

テンプレート:
  8. ナレッジベースのガイド文定義（YAML/Markdown）
  9. 汎用テンプレートの初期セット作成

連携:
  10. F-003 → F-004 トリガー（verdict = "eligible" で自動生成）
  11. F-002 再読解 → チェックリスト再生成（手動チェック保持付き UPSERT）
  12. 進捗管理API（チェック操作・手動追加）
```
