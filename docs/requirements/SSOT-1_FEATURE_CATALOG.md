## 2. 機能間の依存関係

```
F-001 案件自動収集
  │
  ├──→ F-002 AI読解（F-001が収集した公告・仕様書を入力とする）
  │      │
  │      ├──→ F-003 参加可否判定（F-002の案件カードを入力とする）
  │      │
  │      └──→ F-004 チェックリスト生成（F-002の案件カードを入力とする）
  │
  └──→ F-005 価格分析（F-001と独立。落札実績ODを直接取り込む）

F-005 価格分析
  │
  └──→ F-001 スコアリング（F-005の相場データがF-001のスコア計算に使われる）

---

P1以降の依存:
F-002 → F-006 同等品検索（AI読解で抽出した物品スペックが入力）
F-002 → F-007 入札書テンプレ（AI読解で抽出した発注機関情報が入力）
F-003 + F-005 → F-009 学習ループ（判定結果＋価格データが学習の入力）
F-009 → F-001 スコアリング改善（学習結果がスコアに反映）
```

## 3. 機能概要（P0）

### 3.1 F-001: 案件自動収集＋AIフィルタリング

| 項目 | 内容 |
|------|------|
| **目的** | 対象3領域の入札案件を自動で見つけ、スコア付きで提示する |
| **入力** | 検索条件（キーワード、領域、地域、等級） |
| **出力** | スコア付き案件カード一覧 |
| **主要処理** | ① サイト巡回＋データ取得 ② 正規化 ③ フィルタリング ④ スコアリング |
| **対象領域** | ① 軽運送・配送 ② 物品調達 ③ 内装・雑務系役務 |
| **依存** | F-005（相場データ）をスコアリングに使用 |
| **詳細仕様** | [F-001_案件自動収集.md](../design/features/project/F-001_案件自動収集.md) |

### 3.2 F-002: 公告・仕様書のAI読解＋案件カード化

| 項目 | 内容 |
|------|------|
| **目的** | 公告・仕様書を自動で読み解き、判断に必要な情報を構造化する |
| **入力** | 公告HTML/PDF、仕様書PDF |
| **出力** | 案件カード（要点・参加条件・提出物・期限・リスク＋根拠） |
| **主要処理** | ① 文書テキスト抽出 ② LLMで構造化読解 ③ 根拠箇所のマッピング |
| **AI原則** | 断定しない。根拠保持。人間確認前提 |
| **依存** | F-001（収集した文書が入力） |
| **詳細仕様** | [F-002_AI読解.md](../design/features/project/F-002_AI読解.md) |

### 3.3 F-003: 参加可否の自動判定

| 項目 | 内容 |
|------|------|
| **目的** | 自分の資格で参加できる案件かを自動で判定する |
| **入力** | 案件カード（F-002出力）＋ユーザー資格情報 |
| **出力** | 判定結果（参加可/不可＋理由＋根拠） |
| **主要処理** | ルールベース判定（5条件チェック） |
| **判定条件** | ① 全省庁統一資格 ② 等級 ③ 営業品目 ④ 地域 ⑤ 許認可 |
| **依存** | F-002（案件カードが入力） |
| **詳細仕様** | [F-003_参加可否判定.md](../design/features/project/F-003_参加可否判定.md) |

### 3.4 F-004: チェックリスト＋応札準備テンプレ自動生成

| 項目 | 内容 |
|------|------|
| **目的** | 入札に必要な準備を漏れなく期限内に完了するためのガイド |
| **入力** | 案件カード（F-002出力） |
| **出力** | チェックリスト（提出物・期限・ステータス・テンプレリンク） |
| **主要処理** | ① 提出物リスト抽出 ② テンプレ適用 ③ 期限逆算スケジュール生成 |
| **知識ベース** | ホットプラスのノウハウ（入札書記入・封筒ルール・下見積もり書等） |
| **依存** | F-002（案件カードが入力） |
| **詳細仕様** | [F-004_チェックリスト生成.md](../design/features/project/F-004_チェックリスト生成.md) |

### 3.5 F-005: 落札実績データの取り込み＋価格分析

| 項目 | 内容 |
|------|------|
| **目的** | 入札価格の相場感を提供し、適正価格での応札を支援する |
| **入力** | 調達ポータル落札実績オープンデータ（CSV/JSON） |
| **出力** | 価格分析レポート（相場帯・参加社数分布・勝率推定） |
| **主要処理** | ① OD定期取得 ② 正規化・DB格納 ③ 対象領域の集計・分析 |
| **データソース** | 調達ポータル（全件/差分） |
| **依存** | なし（独立して動作。出力はF-001のスコアリングに使用される） |
| **詳細仕様** | [F-005_価格分析.md](../design/features/project/F-005_価格分析.md) |

## 4. 共通要件（全機能に適用）

### 4.1 データ取得・信頼性

| 要件 | 内容 |
|------|------|
| 原本保存 | 取得した公告HTML/PDF/CSVの原本を必ず保存 |
| 根拠保持 | AI抽出結果は根拠箇所（URL/ページ/見出し）を必ず保持 |
| 差分検知 | 更新を検知し差分のみ取得 |
| 重複排除 | 同一案件の重複登録を防止 |
| 取得失敗対応 | リトライ（最大3回）、失敗時アラート |

### 4.2 AI安全性

| 要件 | 内容 |
|------|------|
| 断定しない | 「〜と記載されています」等の表現 |
| 原文確認導線 | 原文へのリンクを必ず提供 |
| ハルシネーション対策 | 根拠なしの情報は「根拠なし」と明示 |
| 人間確認前提 | AI判定を最終判断としない |

### 4.3 Phase1 制約

| 制約 | 内容 |
|------|------|
| ユーザー数 | 1人（マルチテナント不要） |
| UI品質 | 自分が使えれば十分（デザインの作り込みは不要） |
| テスト | 自分が毎日使うことが最大のテスト |
| エラーハンドリング | 致命的エラーはアラート。軽微なエラーはログ |

## 5. 開発ロードマップ

### 5.1 推奨開発順序

| 順序 | 機能 | 理由 |
|------|------|------|
| 1 | F-005 価格分析 | 依存なし。調達ポータルODの取り込みはPhase0検証の延長。データ基盤として先に構築 |
| 2 | F-001 案件自動収集 | F-005の相場データをスコアリングに利用。データ取得パイプラインの構築 |
| 3 | F-002 AI読解 | F-001の収集結果を入力。LLMプロンプト設計が肝 |
| 4 | F-003 参加可否判定 | F-002の案件カードを入力。ルールベースで比較的シンプル |
| 5 | F-004 チェックリスト生成 | F-002の案件カードを入力。ホットプラスのノウハウのルール化 |

### 5.2 Phase0 → Phase1 の境界

```
Phase0（検証）:
  - F-005 の「調達ポータルOD取り込み」部分
  - F-002 の「AI読解精度PoC」部分
  - 対象3領域の案件件数調査

Phase1（本開発）:
  - 全P0機能（F-001〜F-005）のフル実装
  - ダッシュボード（案件一覧・カード・チェックリスト）
  - 日次バッチ（案件収集・データ更新）
```
