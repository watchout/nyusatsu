## 1. GTM 全体戦略

### 1.1 Phase ごとの展開マップ

```
Phase1: 自分で使う → 実績を証明する
  │
  │  「月2件落札」「時短10分」「不備ゼロ」の数字が出る
  │
  ▼
Phase2: 同じ課題を持つ人に売る
  │
  │  ホットプラスコミュニティ経由で「コンサルなしで落札」の体験談を展開
  │  Starter → Pro のアップセルで LTV を伸ばす
  │
  ▼
Phase3: コンサル会社にツールごと売る
     JV モデル（4:6）で、ホットプラスのビジネスを入札ラクダAI で置き換え
```

### 1.2 Phase サマリ

| Phase | 目的 | 対象ペルソナ | 主チャネル | 成功基準 |
|-------|------|-----------|---------|---------|
| 1 | 成果実証 | A（金子・自分） | 自社利用 | 月2件落札 + 毎朝10分運用 + 不備ゼロ |
| 2 | 初期ユーザー獲得 | B（佐藤・入札初心者） | ホットプラスコミュニティ | 有料10社 + チャーン月5%以下 |
| 3 | スケール | C（田中・コンサル企業） | JV提案 | 按分モデル稼働 + 6ヶ月黒字化 |

### 1.3 ポジショニングの一貫性

```
GTM 全体を通して守るメッセージ:

  「入札情報サービス」ではなく「入札オペレーションOS」。
  NJSSと同じ土俵で戦わない。「探した後」で勝つ。

Phase1: 「自分の入札事業を仕組みで回す」
Phase2: 「コンサルなしで入札を回せるツール」
Phase3: 「コンサル会社の工数を半分にするOS」

→ Phase が変わっても「実務の半自動化」という軸はブレない。
```

## 2. Phase1: 自社利用 → 成果実証（A向け導線）

> Phase1 の GTM は「売る」ではなく「証明する」。
> 自分が毎日使って実績を出し、Phase2 の武器にする。

### 2.1 導入ステップ

| Step | やること | 所要時間 | 完了条件 |
|------|---------|---------|---------|
| 1 | セットアップ | 1日 | company_profile 登録 + 検索条件（3領域）設定完了 |
| 2 | 毎朝10分ルーティンの開始 | 2週目〜 | ダッシュボードを毎朝開く習慣が定着 |
| 3 | 初回応札（1件） | 1ヶ月目 | 案件カード→チェックリスト→提出完了。成功/失注を記録 |
| 4 | 応札サイクルの安定化 | 2〜3ヶ月目 | 月5件以上の応札。運用フローの改善ループが回る |
| 5 | 目標 KPI 達成 | 6ヶ月目 | 月10件応札 + 月2件落札 + 不備ゼロ |
| 6 | 実績レポート作成 | 6ヶ月目 | Phase2 の営業材料として Before/After を文書化 |

### 2.2 計測する実績

> PRD §7.2 で定義済みの KPI をそのまま使う。

**事業成果指標**:

| 指標 | 目標 | 計測方法 |
|------|------|---------|
| 応札件数 | 月10件以上 | 案件ステータス（planned 以降）で集計 |
| 落札件数 | 月2件以上 | 案件ステータスで計測 |
| 落札による月間売上 | [要確認] | 落札金額の合計 |

**運用効率指標**:

| 指標 | 目標 | 計測方法 |
|------|------|---------|
| 案件探索時間 | 1日10分以内 | 自己計測 |
| 1案件あたり準備時間 | 7分以内 | バッチログ＋操作ログ |
| 提出不備率 | 0%（5カテゴリ全て） | 入札結果で確認 |

**AI品質指標**:

| 指標 | 目標 | 計測方法 |
|------|------|---------|
| uncertain率 | 20%以下 | case_cards の confidence < 0.6 のアイテム割合 |
| evidence付与率 | 90%以上 | case_cards の根拠箇所有無集計 |

### 2.3 成果実証の形式（Phase2 への武器）

Phase1 で蓄積した実績を、以下の形式で Phase2 の営業材料にする。

**① Before/After レポート**

VP §3.3 のテーブルに **実数値** を埋めたもの:

```
| 工程              | Before（手作業）| After（実績値） | 削減率（実績） |
|-------------------|----------------|----------------|---------------|
| 案件探索           | 120分/日       | 実績: X分       | X%            |
| 仕様書読解         | 240分/件       | 実績: X分       | X%            |
| 参加可否判断       | 30分/件        | 実績: X分       | X%            |
| ...               | ...            | ...             | ...           |
| 合計（1案件あたり） | 5〜9時間       | 実績: X分       | X%            |

→ 仮説の「7分」が実績でどうなったかを検証
```

**② 不備ゼロの証明**

VP §3.4 の5カテゴリに対して実績を記入:

```
| カテゴリ            | 6ヶ月間の実績  |
|---------------------|---------------|
| 提出書類の漏れ       | X件 / Y件応札 |
| 参加資格の見落とし    | X件 / Y件応札 |
| 提出期限超過         | X件 / Y件応札 |
| 参加条件の誤読       | X件 / Y件応札 |
| 価格データ未確認     | X件 / Y件応札 |

→ 「全カテゴリ 0件/N件応札」を目指す
```

**③ ダッシュボード・スクリーンショット集**

| 画面 | 見せる価値 |
|------|---------|
| 案件一覧（スコア順） | 「毎朝これを見るだけで案件チェック完了」 |
| 案件カード（AI読解結果） | 「仕様書を2分で理解できる」 |
| 参加可否判定結果 | 「自動で参加条件をチェック」 |
| チェックリスト＋逆算スケジュール | 「漏れなく準備できる」 |
| 価格分析（落札実績表示） | 「相場を見て適正価格で入札」 |

### 2.4 Phase1 → Phase2 Go/No-Go Gate

> Phase2（SaaS展開）に進む前に、Phase1 で以下の **全基準** を満たすことを必須とする。
> 基準は PRD §7.2 の Phase1 KPI から導出。新しい概念は追加しない。

| # | 基準 | 目標値 | 計測方法 | PRD参照 |
|---|------|--------|---------|---------|
| 1 | 日次10分運用の安定 | 連続30日以上 | 日次運用ログ（朝のチェック〜判断完了の所要時間） | PRD §3.4 |
| 2 | 不備ゼロ（定義済み範囲）達成率 | 80%以上 | 5カテゴリ不備発生率の逆数 | PRD §3.5 |
| 3 | 応札件数 | 月10件以上 | cases ステータス（planned 以降）で集計 | PRD §7.2 |
| 4 | Before/After 実績レポート | 3本以上作成 | §2.3 形式の作成済みレポート数 | GTM §2.3 |

**判定プロセス**: 4基準 **全て** を満たした月末に Go/No-Go レビューを実施。1項目でも未達なら Phase1 を継続。

## 3. Phase2: SaaS展開 → 初期ユーザー獲得（B向け導線）

> Phase2 の GTM は「体験談ベース」。
> A の実績をそのまま B に見せる。

### 3.1 展開チャネル

| # | チャネル | リーチ | コスト | 優先度 | 理由 |
|---|---------|-------|-------|--------|------|
| 1 | ホットプラス会員コミュニティ | 中 | 低 | ★★★ | B が既にいる場所。信頼関係あり |
| 2 | 入札セミナー・勉強会 | 中 | 中 | ★★ | 入札に興味がある人が集まる |
| 3 | SEO（「入札 初心者」「入札 やり方」等） | 低→高 | 低 | ★★ | 長期的に効く。Phase1 実績をブログ化 |
| 4 | 口コミ（B→B 紹介） | 低→高 | ゼロ | ★ | 落札成功者の紹介が最強の営業 |

**Phase2 初期のメインチャネル**: ホットプラス会員コミュニティ

```
理由:
- B（佐藤）が既にいる場所。新規チャネル開拓不要
- A（金子）が会員として信頼関係を構築済み
- 「同じ会員が作ったツールで落札できた」は最も響くメッセージ
- コミュニティ内での紹介→無料トライアル→有料転換の流れが自然
```

### 3.3 実績の見せ方

Phase1 の実績を、B に刺さる形で翻訳する。

| 実績データ（Phase1） | B向けの見せ方 | 使う場面 |
|---------------------|-------------|---------|
| 案件探索 120分→0分 | 「もう毎朝データウェアネットを開かなくていい」 | LP / セミナー冒頭 |
| 仕様書読解 4時間→2分 | 「仕様書を読まなくても、参加条件がわかる」 | 無料トライアルのデモ |
| 不備ゼロ（6ヶ月・X件応札） | 「提出漏れで落選する心配がなくなった」 | 成功事例ページ |
| 月2件落札 | 「コンサルなしで月2件落札。これが自分だけでできた。」 | 口コミテンプレ |
| uncertain率 20%以下 | 「AI読解の信頼度80%以上。根拠箇所も表示。」 | FAQ / 不安解消コンテンツ |

**B に響くメッセージ（USER_PERSONA.md より）**:

```
✅ 響く: 「コンサルなしで、初めての落札。」
✅ 響く: 結果ベース（「月X件落札」「不備ゼロ」「120分→0分」）
✅ 響く: 体験談・同業者の成功事例
❌ 響かない: 「AIが自動で〜」（技術アピールは B に刺さらない）
❌ 響かない: 「入札のプロになろう」（プロになりたいのではなく、楽に回したい）
```

### 3.4 価格導線

> 詳細: VALUE_PROPOSITION.md §3.5

```
■ 導入パス（B のスタンダード）:

  無料トライアル（1案件分）
    ↓
  Starter ¥9,800〜14,800/月
  （案件カード＋チェックリスト＋参加可否判定）
    ↓  初回落札成功
  Pro ¥29,800〜49,800/月
  （+ 失注学習＋テンプレ生成＋外注連携）

■ アップセルのトリガー:
  - 月3件以上応札するようになった → Pro の「テンプレ自動生成」が必要に
  - 失注が続いた → Pro の「失注学習ループ」が欲しくなる
  - 下請けに仕事を回したい → Pro の「外注連携」が必要

■ LTV を伸ばすメカニズム:
  - データ蓄積（落札/失注データ）→ スイッチングコスト上昇
  - スコアリング精度の向上 → 使うほど賢くなる実感
  - 業務フローの定着 → 「これなしでは入札できない」状態
```

## 4. Phase3: コンサルJV → スケール（C向け導線）

> Phase3 の GTM は「ツール提供」。
> コンサル会社の業務基盤として入り込む。

### 4.1 JV 提案の導線

| Step | やること | タイミング | 完了条件 |
|------|---------|-----------|---------|
| 1 | Phase2 で Starter/Pro の実績蓄積 | Phase2 安定後 | 有料10社以上 + チャーン5%以下 |
| 2 | ホットプラス（または類似コンサル企業）に JV 提案 | Phase2 安定後 | 提案書＋デモ＋実績レポート提示 |
| 3 | Biz プラン（¥98,000〜/月）でツール提供 | JV 合意後 | マルチアカウント＋横断ダッシュボード＋監査/権限 |
| 4 | 按分モデル（4:6 = JV先:弊社）で収益分配 | 運用開始後 | 6ヶ月間の試験運用 → 本契約 |

### 4.2 C向け価値の見せ方

| C の痛み（USER_PERSONA.md より） | 解決策 | 見せ方 |
|-------------------------------|-------|-------|
| 会員ごとに毎回ゼロから案件確認 | AI読解＋案件カードの事前自動生成 | 「会員への提案が自動で用意される」 |
| 同じ説明を何度も繰り返す | チェックリスト標準化＋用語解説AI | 「説明の属人化がなくなる」 |
| 対応品質がバラバラ（属人化） | 横断ダッシュボード＋承認フロー | 「全会員の状況が一画面で見える」 |

**C に響くメッセージ**:
```
✅ 「会員1人あたりの対応工数を50%削減」
✅ 「担当会員数を2倍に増やせる」
✅ 「対応品質の均一化（チェックリスト標準化）」
❌ 「AIツールです」（C が欲しいのはツールではなく工数削減）
```

## 5. メッセージ戦略

## 6. リスクと対策

| # | リスク | 影響度 | 対策 |
|---|-------|-------|------|
| 1 | Phase1 で月2件落札できない → 実績がない | 高 | 対象領域を3つに分散。1件でも落札すれば「初めての落札」として体験談化 |
| 2 | ホットプラスコミュニティでの反応が薄い | 中 | 入札セミナー・SEO・SNS を並行で準備。1チャネル依存しない |
| 3 | B のトライアル→有料転換率が低い | 中 | トライアルで「AI読解＋チェックリスト」まで体験させる。1案件丸ごとの体験が鍵 |
| 4 | Pro へのアップセルが進まない | 中 | 失注が起きたタイミングで「失注学習ループ」の価値をプッシュ通知 |
| 5 | JV 先（ホットプラス等）が入札ラクダAI を競合と見なす | 高 | 「競合」ではなく「ツール提供」として提案。既存ビジネスの補完であることを強調 |

## 7. 検証計画

### 7.1 Phase ごとの検証

| Phase | 検証する仮説 | 検証方法 | 成功基準 | 失敗時のアクション |
|-------|------------|---------|---------|-----------------|
| 1 | 自社運用で実績が出るか | PRD §7.2 の KPI 計測 | 月2件落札 + 時短 + 不備ゼロ | 対象領域の変更 or AI読解の精度改善 |
| 2 | B が有料契約するか | ホットプラスコミュニティでβ提供 | 10社契約 | 価格の見直し or トライアル体験の改善 |
| 2 | B の初期体験が成功するか | トライアル→初回落札の転換率 | 30%以上 | オンボーディングフロー改善 |
| 2 | Starter→Pro のアップセルが回るか | Pro 転換率 | [要確認] | アップセルトリガーの見直し |
| 3 | JV モデルが成立するか | コンサル1社との試験導入 | 6ヶ月間運用 + 黒字化 | 按分比率の見直し or 別モデル検討 |

### 7.2 Phase2 で計測すべき追加指標

| 指標 | 定義 | 目標 |
|------|------|------|
