## 2. 課題（Problem）

### 2.1 解決したい課題

**主要課題（1つだけ）**:
```
入札は儲かるのに、探す・読む・手続きの手間が膨大すぎて、
本業が忙しい経営者が参入・継続できない。
```

**関連する課題**:
1. 用語・手順がわからず、公告・仕様書を読んでも判断できない（初心者の壁）
2. 案件情報が分散しており、おいしい案件を探しきれない（探索地獄）
3. 提出物の漏れ・形式不備・期限切れで無効/失注になるリスクが高い（手続き地獄）

### 2.2 課題の深掘り

| 質問 | 回答 |
|------|------|
| この課題は誰が持っている？ | 統一資格を取ったが入札経験がない中小企業経営者（自分自身） |
| どんな状況で発生する？ | 入札を事業に組み込もうとした瞬間。案件探しから仕様書読解まで全てが壁 |
| どのくらいの頻度で発生する？ | 毎日（案件は日々更新される） |
| 発生すると何が困る？ | 入札を諦める or 膨大な時間を取られて本業に支障 |
| 今はどう対処している？ | ホットプラス（コンサル）に会費を払い、サポートを受けている |
| 対処にいくらかかっている？（時間・お金） | ホットプラス月額会費＋自分の調査時間（1日1〜2時間想定） |

### 2.3 課題の検証状況

| 検証方法 | 実施 | 結果 |
|---------|------|------|
| 自分自身が課題を経験 | ☑ | 統一資格は取得したが、入札の複雑さに阻まれている |
| 身近な人にヒアリング（5人以上） | ☐ | [要確認] Phase1で検証予定 |
| SNS/コミュニティでの反応確認 | ☐ | [要確認] ホットプラスコミュニティ等で確認予定 |
| 既存サービスのレビュー分析 | ☑ | NJSS/入札王等は「探す」は強いが「実務自動化」は未対応 |
| 検索ボリューム確認 | ☐ | [要確認] 「入札 初心者」「入札 やり方」等で確認予定 |

## 3. ソリューション（Solution）

### 3.1 提供する解決策

```
入札の「探す→読む→判断する→準備する」を半自動化し、
本業が忙しくても元請けとして入札事業を回せるようにする。
ホットプラスのノウハウをAIに吸収させ、初心者でも迷わず進める仕組み。
```

### 3.2 主要機能（MVP）

| # | 機能 | 課題との関連 | 優先度 |
|---|------|-------------|-------|
| 1 | 案件自動収集＋AIフィルタリング | 探索地獄の解消。軽運送・物品・内装に合う案件を自動で収集・スコアリング | P0 |
| 2 | 公告・仕様書のAI読解＋案件カード化 | 読解地獄の解消。要点・参加条件・提出物・期限・リスクを自動抽出 | P0 |
| 3 | 参加可否の自動判定 | 参加条件（統一資格・等級・営業品目・地域）を自動チェック | P0 |
| 4 | チェックリスト＋応札準備テンプレ自動生成 | 手続き地獄の解消。提出物リスト・期限・テンプレートを自動生成 | P0 |
| 5 | 落札実績データの取り込み＋価格分析 | 値決めの根拠。過去の落札金額・参加社数で勝率を推定 | P0 |

**注意**: MVP は 3-5 機能に絞る

### 3.3 将来の機能（Phase 2以降）

| # | 機能 | 実装時期 |
|---|------|---------|
| 1 | 同等品の自動検索＋比較表生成（物品案件の原価圧縮） | Phase 2 |
| 2 | 入札書・封筒テンプレートの自動生成 | Phase 2 |
| 3 | 失注分析＋改善提案（学習ループ） | Phase 2 |
| 4 | 落札後の業務管理（工程・証跡・報告書・請求書） | Phase 3 |
| 5 | マルチアカウント対応＋ダッシュボード（SaaS化） | Phase 3 |

## 4. ターゲットユーザー（Target User）

### 4.1 プライマリターゲット

| 項目 | 内容 |
|------|------|
| 誰？ | 自分自身（入札を事業化したい中小企業経営者） |
| 年齢層 | 30〜50代 |
| 職業/役割 | 複数事業を兼業する経営者。弱電・IT・軽運送等のネットワークを持つ |
| 課題の深刻度 | 高（統一資格を取ったのに活用できていない＝機会損失） |
| 支払い意思 | 高（Phase1は自社ツールなので開発コストのみ。落札利益で回収） |
| リーチしやすさ | 高（自分自身がユーザー） |

### 4.2 セカンダリターゲット

| 項目 | 内容 |
|------|------|
| 誰？ | ホットプラス会員のような入札初心者（統一資格取得済み、まだ経験が浅い） |
| プライマリとの違い | ITリテラシーがバラつく。コンサルのサポートに依存している |

### 4.3 ターゲットではない人

| 対象外 | 理由 |
|-------|------|
| 大企業の入札専任部門 | 既に独自のフローとシステムがあり、ツールの必要性が低い |
| 建設業の指名競争入札が中心の企業 | 全省庁統一資格（一般競争入札）とはフローが異なる |

## 5. 差別化（Differentiation）

### 5.1 なぜあなたが作るのか？

| 質問 | 回答 |
|------|------|
| この領域の経験・知識 | 統一資格取得済み。ホットプラス会員としてノウハウを学習中。弱電・IT・軽運送のネットワークあり |
| ユニークな視点・アプローチ | IT/AIエンジニアとして「自動化」が本業。入札の手作業を技術で消す発想 |
| 他の人にない強み | 自分がユーザーかつ開発者。下請けネットワーク（義父:軽運送、知人:内装・電気）を持つ |
| なぜ今このタイミング？ | LLMの進化で仕様書の自動読解が現実的になった。入札情報のオープンデータ化も進行中 |

### 5.2 競合との違い

| 比較軸 | NJSS | 入札王 | 入札ラクダAI |
|-------|------|-------|-------------|
| 主要機能 | 案件検索・AIレコメンド・案件管理 | 新着配信・検索・予算情報 | **案件探索→読解→応札準備の半自動化** |
| 価格 | 月額制（法人向け） | 月額制（法人向け） | Phase1: 自社利用のみ（開発コスト） |
| ターゲット | 入札担当者全般 | 入札担当者全般 | **入札初心者・兼業経営者** |
| **差別化ポイント** | 探すまで | 探すまで | **「探す→読む→準備する」まで一気通貫** |

### 5.3 参入障壁（Moat）

```
競合が真似しにくい理由:
1. ホットプラスのノウハウをAIに吸収させた「入札判断エンジン」（独自知識ベース）
2. 自分自身がユーザーとして毎日使い、フィードバックループが最速で回る
3. 下請けネットワーク（軽運送・内装・電気）との連携データが蓄積される
4. 落札/失注データの蓄積による勝ちパターン学習（使うほど賢くなる）
```

## 8. リスクと前提

### 8.1 主要なリスク

| リスク | 影響度 | 対策 |
|-------|-------|------|
| データ取得の網羅性・正確性（分散サイト・PDF・構造変更） | 高 | 差分検知・重複排除・原文保存・失敗時復旧をMVPから必須 |
| AI読解の精度（仕様書の誤読 → 無効入札・機会損失） | 高 | 必ず原文リンク・根拠箇所を保持。要約は断定しない設計 |
| 初めての入札で落札できない → モチベーション低下 | 中 | 失注分析＋改善提案で学習ループを回す |

### 8.2 検証すべき前提

| 前提 | 検証方法 | 検証時期 |
|------|---------|---------|
| 軽運送・物品・内装の案件が十分な件数存在するか | データウェアネット＋調達ポータルでの件数調査 | Phase 0 |
| AI（LLM）で仕様書を実用レベルで読解できるか | 実際の仕様書PDFでのPoC | Phase 0 |
| 下請けネットワーク（義父・知人）で実際に案件を回せるか | 最初の1件の落札で検証 | Phase 1 |

### 10.2 Go/No-Go 判断基準

**Go条件**:
- [x] 自分自身が課題を経験している（統一資格取得済みだが活用できていない）
- [ ] 対象領域の案件が月20件以上存在する
