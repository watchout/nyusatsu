# 入札ラクダAI（仮）— Claude Code 開発README（v1）
作成日: 2026-02-13

## 1. 目的
官公庁・自治体等の入札業務を、**「わかる・見つかる・進められる」**状態に標準化し、  
案件探索→選定→入札準備→入札→結果学習→落札後（実行・報告・請求）までを **半自動〜自動化** する。

## 2. 対象ユーザー（最初のMVPターゲット）
- 中小企業の兼任入札担当（初心者〜1年目）
- 入札コンサル/運用代行（例：ホットプラス想定）
- 落札後を回す現場PM（協力会社・下請け調整）

## 3. MVPの成功条件（測れる指標）
- **探索時間**：1日あたり XX分→YY分（例：60→10）
- **入札準備の手戻り**（提出物漏れ・差戻し）：XX%減
- **案件取り漏れ**：0に近づける（通知・保存検索・差分検知）
- **落札後の事務時間**（報告/請求）：XX%減
- **学習ループ**：失注理由タグ付け率 90%+、次回改善提案生成 80%+

## 5. スコープ（Phase分割）
### Phase 0: 公式データの確実取得 + 案件カード化
- 収集：調達ポータル（落札実績OD）／統合PPI／防衛省系（分散サイト）など
- 正規化：発注機関、案件名、区分（物品/役務/工事）、地域、期限、添付
- 案件カード：要点、提出物、期限、NG条件、リスク、根拠（原文リンク/引用箇所）

### Phase 1: 案件選定スコア + チェックリスト駆動の入札準備
- スコア：期待粗利、工数、移動、勝ち筋（過去落札傾向）、リスク
- チェックリスト：提出物テンプレ、期日、社内タスク、外注見積依頼

### Phase 2: 下請け/協力会社アサイン + 落札後の実行管理
- 協力会社マスタ：対応範囲、単価、稼働、エリア、評価
- 受注後テンプレ：工程、証跡（写真/検収）、報告書、請求書生成

### Phase 3: アカウント横断の学習・ダッシュボード
- 失注理由 taxonomy、改善提案、勝ちパターン蓄積
- マスターダッシュボード（全クライアント統合）

## 6. ディレクトリ案（例）
```
/docs
  /01_painbook.md
  /02_persona_stories.md
  /03_differentiation.md
  /04_ssot.md
/app
  /collectors        # 取得（HTML/PDF/CSV/JSON）
  /normalizers       # 正規化
  /extractors        # 仕様書抽出（RAG/要約/根拠）
  /scorers           # 案件選定スコア
  /workflows         # チェックリスト/タスク
  /reporting         # 報告書/請求書
  /dashboards        # 分析
```

## 7. Claude Code に渡す実装ルール（最小）
- **原文保存**：公告HTML/PDF/CSVの原本を必ず保存（監査・再現性）
- **根拠提示**：抽出結果は、必ず「根拠箇所（URL/ページ/見出し）」を保持
- **安全設計**：要約は断定しない（原文確認導線を必須）
- **失敗前提**：取得失敗・構造変更・重複を前提に、リトライ/差分/重複排除を実装
