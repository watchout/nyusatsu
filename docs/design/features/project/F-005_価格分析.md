## 2. ユーザーストーリー

### メインストーリー

> **ペルソナ1（金子）として、**
> 対象領域（軽運送・物品・内装）の過去の落札データを見て、
> 「この領域の案件はだいたい○○万円〜○○万円で落札されている」
> 「参加社数は平均○社で、1社入札の割合は○%」
> とわかれば、入札価格を決めるときの判断材料になる。

### 具体シナリオ

**シナリオ1: 初回利用 — 相場を把握する**
1. システムが調達ポータルから落札実績オープンデータを取り込み済み
2. 金子が「軽運送」で検索すると、過去3年間の落札データが表示される
3. 落札金額の中央値: 120万円、四分位範囲: 80万〜200万円
4. 参加社数の平均: 3.2社、1社入札の割合: 35%
5. → 「意外と競争が少ない。120万円前後が相場か」と判断できる

**シナリオ2: 案件ごとの価格判断**
1. F-001で見つけた案件「○○省 軽運送業務」の入札を検討中
2. 発注機関「○○省」× 領域「軽運送」で絞り込む
3. 同じ発注機関の過去落札データ: 100万円〜150万円、参加社数: 2〜4社
4. → 「この機関は100万前後で入れれば勝てそう」と判断材料になる

**シナリオ3: データの自動更新**
1. 週次で差分データが自動取得される
2. 新しい落札結果が反映され、トレンドが更新される
3. 金子は常に最新の相場感を持てる

## 3. 機能要件

### 3-A. 入力

#### 3-A-1. データソース（2層構造）

| 層 | データソース | 形式 | 取得方法 | 取得内容 | 備考 |
|----|------------|------|---------|---------|------|
| Layer 1: 一括取得 | 調達ポータル 落札実績OD | CSV（UTF-8 BOM付き） | HTTPダウンロード（ZIP） | 落札金額、落札者、発注機関、案件名、開札日 等 | 全件＋差分ファイルが公開。ベースデータとして大量一括取得 |
| Layer 2: 補完取得 | 調達ポータル 落札公告詳細ページ | HTML | スクレイピング | **参加社数**、**予定価格（事後公表分）**、各応札者の入札金額 | 対象領域に絞って詳細取得。ODで不足するデータを補完 |

```
Layer 1（OD一括） = 全案件のベースデータ（金額・機関・日付等）
      ↓
Layer 2（公告補完） = 対象領域の案件のみ、参加社数・予定価格・応札者情報を補完
```

> **[要確認] 調達ポータルODの実際のCSVスキーマ**
> 実際にダウンロードして、カラム名・データ型・区切り文字を確認する必要がある。
> Phase0（M6）の検証タスクとして実施予定。
> ファイル仕様PDF: https://www.p-portal.go.jp/pps-web-biz/resources/app/pdf/rakusatsuopendata.pdf

#### 3-A-2. ユーザー入力（設定）

| 項目 | 内容 | デフォルト |
|------|------|----------|
| 対象領域キーワード | 分析対象の領域を特定するキーワード群 | 「軽運送」「配送」「物品」「清掃」「内装」等 |
| 分析対象期間 | 集計の対象期間 | 直近3年 |
| 発注機関フィルタ | 特定の発注機関に絞り込み | 全機関 |

### 3-B. 処理

#### 3-B-1. データ取得パイプライン（2層構造）

```
═══════════════════════════════════════════════
  Layer 1: OD一括取得（全案件のベースデータ）
═══════════════════════════════════════════════

[調達ポータル OD (CSV/ZIP)]
       │
       ▼
  ① HTTPダウンロード（全件 or 差分）
       │
       ▼
  ② ファイル検証（サイズ・ハッシュ・行数）
       │
       ▼
  ③ 原本保存（data/raw/od/ にタイムスタンプ付き）
       │
       ▼
  ④ パース・正規化（CSV → 統一スキーマ）
       │
       ▼
  ⑤ DB格納: base_bids テーブル（重複排除 → UPSERT）

═══════════════════════════════════════════════
  Layer 2: 公告詳細ページ補完（対象領域のみ）
═══════════════════════════════════════════════

[base_bids から対象領域レコードを抽出]
       │
       ▼
  ⑥ 落札公告詳細ページURL を生成
       │
       ▼
  ⑦ HTMLスクレイピング
     → 参加社数、予定価格（事後公表）、各応札者の入札金額
       │
       ▼
  ⑧ DB格納: bid_details テーブル（base_bidsに紐付け）
```

**Layer 1 取得ルール:**

| 項目 | ルール |
|------|-------|
| 初回取得 | 全件ファイルをダウンロード |
| 更新取得 | 差分ファイルをダウンロード → 前回取得日時以降の分のみ処理 |
| 差分検知 | ファイルのHTTPヘッダ（Last-Modified / ETag）を利用 |
| リトライ | 取得失敗時、最大3回リトライ（間隔: 30秒→60秒→120秒） |
| 失敗時対応 | 3回失敗でアラート発報。次回定期実行時に再試行 |
| 実行頻度 | 日次（デフォルト）。設定で変更可能 |

**Layer 2 取得ルール:**

| 項目 | ルール |
|------|-------|
| 対象 | Layer 1 で取り込んだレコードのうち、対象領域キーワードに一致する案件のみ |
| 取得タイミング | Layer 1 の取り込み完了後に実行 |
| アクセス間隔 | 1リクエストあたり最低1秒の間隔（サーバー負荷配慮） |
| リトライ | 個別ページの取得失敗は3回リトライ後スキップ。次回バッチで再試行 |
| キャッシュ | 取得済みページは再取得しない（bid_details に記録がある場合スキップ） |
| 実行頻度 | Layer 1 と同時（日次） |

#### 3-B-2. 正規化・格納（2テーブル構造）

**テーブル1: base_bids（Layer 1: OD由来のベースデータ）**

| カラム | 型 | 説明 | 備考 |
|--------|-----|------|------|
| id | UUID | 内部ID | 自動採番（PK） |
| source_id | VARCHAR | ODでの案件識別子 | 重複排除のキー（UNIQUE） |
| case_name | TEXT | 案件名 | |
| issuing_org | VARCHAR | 発注機関名 | |
| issuing_org_code | VARCHAR | 発注機関コード | ODに含まれない場合はNULL。Layer 2 or 名寄せで補完 |
| bid_type | VARCHAR | 入札方式（一般競争/指名等） | |
| category | VARCHAR | 品目分類 | |
| winning_amount | BIGINT | 落札金額（円） | 税抜 or 税込は [要確認] |
| winning_bidder | VARCHAR | 落札者名 | |
| opening_date | DATE | 開札日 | |
| contract_date | DATE | 契約日 | |
| detail_url | TEXT | 落札公告詳細ページのURL | Layer 2 のスクレイピング対象 |
| raw_data | JSONB | OD元データ全体 | スキーマ変更への備え |
| imported_at | TIMESTAMP | 取り込み日時 | |

**テーブル2: bid_details（Layer 2: 公告詳細ページ由来の補完データ）**

| カラム | 型 | 説明 | 備考 |
|--------|-----|------|------|
| id | UUID | 内部ID | 自動採番（PK） |
| base_bid_id | UUID | base_bids.id への FK | |
| num_participants | INTEGER | 参加社数（応札者数） | 公告詳細ページから取得 |
| budget_amount | BIGINT | 予定価格 | 事後公表の場合のみ。非公表はNULL |
| winning_rate | DECIMAL | 落札率（= winning_amount / budget_amount） | budget_amount がある場合に自動計算 |
| bidder_details | JSONB | 各応札者の入札金額等 | `[{"name": "A社", "amount": 1000000}, ...]` |
| raw_html | TEXT | 取得したHTMLの原本 | 再パース用に保存 |
| scraped_at | TIMESTAMP | スクレイピング実行日時 | |

> **[要確認] base_bids のスキーマはODの実際のカラムに基づいて調整が必要。**
> Phase0（M6）で実際のODファイルをダウンロードし、スキーマを確定する。
> `raw_data` にJSONBで全カラムを保存し、スキーマ変更に備える。

**DB: PostgreSQL（確定）**
- JSONB対応（raw_data, bidder_details）
- ウィンドウ関数・CTE で集計クエリが書きやすい
- asyncpg で FastAPI と相性良好

**重複排除ルール:**
- `source_id`（ODの案件識別子）をキーに重複チェック
- 同一 `source_id` が存在する場合、`opening_date` が新しい方で UPSERT
- 重複検出時はログに記録（INFO レベル）
- bid_details は `base_bid_id` ごとに1レコード。再取得時は UPSERT

#### 3-B-3. 分析・集計

**対象領域フィルタリング:**
- ユーザー設定の「対象領域キーワード」で `case_name` を部分一致検索
- 複数キーワードは OR 条件
- キーワード辞書は将来的に拡張可能な設計とする

**集計項目:**

| # | 集計 | 算出方法 | データソース | 単位 |
|---|------|---------|------------|------|
| 1 | 落札金額の分布 | 最小値、Q1、中央値、Q3、最大値 | Layer 1（base_bids） | 円 |
| 2 | 落札金額の平均 | 算術平均（外れ値除外オプション付き） | Layer 1（base_bids） | 円 |
| 3 | 参加社数の分布 | 最小値、Q1、中央値、Q3、最大値 | **Layer 2（bid_details）** | 社 |
| 4 | 1社入札の割合 | 参加社数=1 の件数 / 全件数 | **Layer 2（bid_details）** | % |
| 5 | 落札率の分布 | winning_amount / budget_amount | **Layer 2（bid_details）** | % |
| 6 | 期間別トレンド | 半年単位で上記指標を算出 | Layer 1 + 2 | 時系列 |
| 7 | 発注機関別集計 | 上記指標を発注機関ごとに算出 | Layer 1 + 2 | — |
| 8 | 案件数推移 | 月別の案件件数 | Layer 1（base_bids） | 件/月 |

**外れ値処理:**
- IQR（四分位範囲）方式: Q1 - 1.5×IQR 未満、Q3 + 1.5×IQR 超を外れ値として除外（集計時）
- 外れ値を含む/含まないの両方の値を保持

### 3-C. 出力

#### 3-C-1. 価格分析レポート

ユーザーが対象領域を指定して閲覧するレポート。

| セクション | 内容 | データソース |
|-----------|------|------------|
| サマリー | 対象期間、対象件数、落札金額の中央値・平均・範囲 | Layer 1 |
| 金額分布 | ヒストグラム用データ（バケット幅は自動調整） | Layer 1 |
| 参加社数分布 | ヒストグラム用データ | **Layer 2** |
| 1社入札率 | 全体の1社入札割合＋領域別の割合 | **Layer 2** |
| 落札率分布 | 落札金額÷予定価格の分布（「何割で入れれば勝てるか」の目安） | **Layer 2** |
| 期間別トレンド | 半年単位の時系列データ（金額中央値、件数、参加社数） | Layer 1 + 2 |
| 発注機関ランキング | 案件数 Top10 の発注機関＋各機関の金額・参加社数・落札率 | Layer 1 + 2 |

#### 3-C-2. 案件スコアリング用データ（F-001向け API/内部データ）

F-001のスコアリングロジックに提供するデータ。

| データ | 内容 | 用途 |
|-------|------|------|
| 領域別案件数（月平均） | 対象キーワードに一致する月平均案件数 | 案件の希少性判定 |
| 領域別落札金額中央値 | 対象領域の落札金額の中央値 | 案件規模の評価 |
| 領域別平均参加社数 | 対象領域の平均参加社数 | 競争度の評価 |
| 発注機関別統計 | 特定機関の落札傾向 | 機関ごとの勝率推定 |

### 3-D. ビジネスルール

| # | ルール | 詳細 |
|---|-------|------|
| 1 | 原本保存 | Layer 1: CSV原本を `data/raw/od/` に保存。Layer 2: HTML原本を `bid_details.raw_html` に保存。削除は手動のみ |
| 2 | データ鮮度 | 最低週1回の更新。デフォルトは日次 |
| 3 | デフォルト分析期間 | 直近3年。ユーザーが変更可能（1年〜全期間） |
| 4 | 金額の扱い | 全て円単位（整数）。税抜 or 税込はODの仕様に準拠 [要確認] |
| 5 | 非公開データ | ODに含まれないデータ（不落随契の内部価格等）は対象外 |
| 6 | データ品質 | 金額が0円または異常値（負の値等）のレコードはフラグを立てて集計から除外 |
| 7 | Layer 2 アクセスマナー | サーバー負荷を考慮し、1リクエスト/秒以下のレート制限を遵守 |
| 8 | 予定価格のNULL許容 | 非公表案件の予定価格はNULL。落札率の集計はNULLを除外して算出 |

### 6-2. 技術選択の方針

| 項目 | 方針 | 備考 |
|------|------|------|
| DB | **PostgreSQL（確定）** | JSONB対応、ウィンドウ関数、asyncpg連携 |
| バッチ実行 | Python スクリプト + cron（Phase1） | 後にCelery等に移行可能 |
| HTTP取得 | httpx（非同期対応） | リトライは tenacity で実装 |
| CSVパース | pandas or polars | データ量次第で選択 |
| HTMLスクレイピング | beautifulsoup4 + httpx | Layer 2 用。レート制限付き |
| 集計 | SQL（DB側）+ Python（後処理） | base_bids + bid_details の JOIN で集計 |

```
  5. 落札公告詳細ページ スクレイパー

```
