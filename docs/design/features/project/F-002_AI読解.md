## 2. ユーザーストーリー

### メインストーリー

> **ペルソナ1（金子）として、**
> F-001で見つけた案件の公告・仕様書をAIが読み解いてくれれば、
> 「参加できるか」「何を提出するか」「リスクは何か」がひと目でわかり、
> 入札の専門知識がなくても案件の判断・準備に進める。

### 具体シナリオ

**シナリオ1: 公告の自動読解 — 参加条件を知りたい**
1. F-001で収集された案件「○○省 軽運送業務委託」を選択
2. AI読解が自動で公告HTMLを解析
3. 案件カードに構造化された参加条件が表示される:
   - 全省庁統一資格: **あり**（根拠: 公告p.1「全省庁統一資格〜を有する者」）
   - 等級: **D等級以上**（根拠: 公告p.1「D等級」）
   - 営業品目: **「その他」で参加可能**（根拠: 公告p.1「物品の販売」）
   - 地域: **関東甲信越**（根拠: 公告p.1「競争参加地域: 関東・甲信越」）
   - 許認可: **不要**
4. → 自分が参加できるかが一瞬でわかる

**シナリオ2: 仕様書PDFの要点抽出 — 業務内容を把握したい**
1. 仕様書PDFが自動でダウンロード・解析される
2. 案件カードに業務内容の要約が表示される:
   - 業務種別: **物品の販売**
   - 品目: 「○○消耗品 500個」「△△備品 10台」
   - 納入場所: **東京都千代田区 ○○庁舎**（1箇所）
   - 納入期限: **2026-04-30**
   - 下見積もり: **要（期限: 2026-03-05）** ← ⚠️ 期限が迫っている
3. → 業務の全体像が把握でき、対応可否を判断できる

**シナリオ3: 提出物リストの抽出 — 何を準備すればいいか知りたい**
1. 案件カードの「提出物」セクションを確認
2. 2種類に分かれて表示される:
   - **入札参加時**: 入札書、委任状、全省庁統一資格の写し
   - **履行時**: 作業者名簿、業務計画書、保険証の写し
3. 各提出物に「テンプレートあり」「発注機関書式」等の補足情報付き
4. → F-004のチェックリスト生成に直接つながる

**シナリオ4: リスクの早期発見 — 落とし穴を見逃さない**
1. 案件カードの「リスク要因」セクションにフラグが立っている:
   - ⚠️ **下見積もり期限が3日後**（対応急ぐ）
   - ⚠️ **複数納入先**（東京3箇所＋大阪1箇所）
   - ⚠️ **Pマーク保有要件あり**
2. → 参加前にリスクを把握し、見送り判断もできる

**シナリオ5: 用語の解説 — わからない言葉を理解する**
1. 案件カードで「技術審査」「総合評価落札方式」等の専門用語が出現
2. ホットプラスのナレッジを参照した用語解説がツールチップで表示される
3. → 入札初心者でも用語の意味を理解できる

### 3-B. 処理

#### 3-B-1. AI読解パイプライン（3段階）

```
═══════════════════════════════════════════════
  Stage 1: 文書取得・テキスト抽出
═══════════════════════════════════════════════

[F-001 cases テーブルから対象案件を取得]
       │
       ▼
  ① 公告HTMLの取得・テキスト抽出
     → beautifulsoup4 でHTMLパース
     → セクション構造（見出し・段落）を保持
       │
       ▼
  ② 仕様書PDFの取得・テキスト抽出
     → pdfplumber でテキスト抽出
     → ページ番号を保持（根拠マッピング用）
     → テーブル構造を検出・保持
     → **スキャンPDF判定**（下記フォールバック戦略参照）
       │
       ▼
  ②-b キャッシュ判定（SHA-256）
     → PDFの SHA-256 ハッシュを計算
     → 同一ハッシュが既にDBに存在する場合、Stage 1 をスキップし既存テキストを再利用
     → 公告HTMLも URL + Content-Hash で同一性を判定
       │
       ▼
  ③ 原本保存
     → HTML: data/raw/notices/{case_id}.html
     → PDF: data/raw/specs/{case_id}.pdf
     → 抽出テキスト: data/raw/texts/{case_id}.txt
     → SHA-256ハッシュ: case_cards.file_hash に記録

═══════════════════════════════════════════════
  Stage 2: LLM構造化抽出
═══════════════════════════════════════════════

[抽出テキスト＋プロンプトテンプレート]
       │
       ▼
  ④ セクション分割（長文対応）
     → 入力テキストのトークン数を計測
     → 5,000トークン超の場合: 見出し単位でチャンク分割 → 並列抽出
     → 5,000トークン以下の場合: 一括抽出
       │
       ▼
  ⑤ LLM API呼び出し（Claude API想定）
     → 構造化JSONで抽出結果を返すよう指示
     → 抽出対象5カテゴリ（下記参照）
     → 各項目に **根拠情報** + **assertion_type**（fact/inferred/caution）を付与するよう指示
     → チャンク分割時: 各チャンクの結果をマージ（重複項目は confidence 高い方を採用）
       │
       ▼
  ⑥ レスポンスのバリデーション
     → JSONスキーマに対するバリデーション
     → 必須項目の欠損チェック
     → 不正な値（日付フォーマット等）の検出
     → assertion_type が fact/inferred/caution のいずれかであることを検証

═══════════════════════════════════════════════
  Stage 3: 根拠マッピング・品質チェック
═══════════════════════════════════════════════

  ⑦ 根拠マッピング（局所探索方式）
     → LLMが返した引用テキスト（quote）を元テキスト内で局所探索
     → 探索方式: 引用テキストを中心に前後50行で近傍マッチング
     → 照合前に正規化（全角→半角、空白統一、改行除去）を適用
     → 二段階閾値で照合:
        ・強一致（Jaccard類似度 ≧ 0.8）: confidence = "high"
        ・候補一致（Jaccard類似度 0.65〜0.8）: confidence = "medium"（UIで「要確認」扱い）
        ・不一致（Jaccard類似度 < 0.65）: confidence = "low"（要人間確認フラグ）
     → 補助手段: Jaccard で候補一致以下の場合、正規化レーベンシュタイン距離でも判定し、
        いずれかの手法で強一致なら confidence = "high" に引き上げ

     **根拠データの固定構造（PDF の場合）:**
     ```json
     {
       "source_type": "pdf",
       "page": 3,
       "section": "第2条 参加資格",
       "quote": "全省庁統一資格を有する者であること",
       "quote_max_chars": 100,
       "assertion_type": "fact"
     }
     ```

     **根拠データの固定構造（HTML の場合）:**
     ```json
     {
       "source_type": "html",
       "selector": "#section-eligibility",
       "heading_path": "入札公告 > 参加資格",
       "quote": "全省庁統一資格を有する者であること",
       "quote_max_chars": 100,
       "assertion_type": "fact"
     }
     ```

     **根拠なし時のルール:**
     - confidence = "low"
     - UIで「要確認」に自動分類
     - assertion_type = "inferred" または "caution" を付与
       │
       ▼
  ⑧ 品質チェック
     → 抽出結果の信頼度スコアを算出
     → 信頼度 = (high項目数 + medium項目数×0.5) / 全項目数
     → assertion_counts を集計: `{"fact": N, "inferred": N, "caution": N}`
     → 信頼度が閾値未満の場合、ログ＋アラート
       │
       ▼
  ⑨ DB格納: case_cards テーブル
```

#### 3-B-1a. スキャンPDF フォールバック戦略

**スキャンPDF判定条件（is_scanned 推定）:**

| # | 条件 | 説明 |
|---|------|------|
| 1 | 抽出文字数 < 50文字/ページ | テキスト層がほぼ存在しない |
| 2 | 記号率 > 60% | 文字種が不自然（化けている） |
| 3 | ページ数に対してテキスト量が極端に少ない | 例: 20ページなのに合計100文字未満 |

いずれか1つでも該当 → `is_scanned = true`

**フォールバック動作:**

| Phase | 動作 | extraction_method | confidence_cap |
|-------|------|-------------------|---------------|
| Phase1 | OCR非対応 → `status='failed'` + 理由ログ | `text_failed` | — |
| Phase2以降 | OCR（Tesseract / Google Vision API）にフォールバック | `ocr` | 0.7（通常は上限なし） |

- `is_scanned` フラグを `case_cards` テーブルに記録 → Phase0の統計集計で画像PDF比率を検証
- OCR結果は通常のテキスト抽出結果と同一スキーマで格納（extraction_method で区別）

#### 3-B-2. LLM構造化抽出の5カテゴリ

**全カテゴリ共通: assertion_type**

全抽出項目に以下の assertion_type を付与する:

| assertion_type | 意味 | UIでの扱い |
|---------------|------|-----------|
| `fact` | テキストに明記されている事実 | そのまま表示 |
| `inferred` | テキストから推定した情報 | ⚠️ マーク付き「推定」ラベル |
| `caution` | 注意喚起（リスク検出・曖昧記載等） | ⚠️ 強調表示 |

LLMプロンプトに assertion_type の返却を指示。バリデーションで `fact`/`inferred`/`caution` のいずれかであることをチェック。

**カテゴリ1: 参加条件（eligibility）**

| 抽出項目 | 型 | 説明 | 根拠ソース | assertion_type 例 |
|---------|-----|------|----------|-----------------|
| unified_qualification | BOOLEAN | 全省庁統一資格の記載有無 | 公告HTML | fact |
| grade | VARCHAR | 等級（A/B/C/D） | 公告HTML | fact |
| business_category | VARCHAR | 営業品目 | 公告HTML | fact / inferred |
| region | VARCHAR | 競争参加地域 | 公告HTML | fact |
| additional_requirements | JSONB | その他の参加要件（許認可、実績、ISO等） | 公告HTML | fact / caution |

> **ナレッジ参照**: KNOWLEDGE_DIGEST STEP2 の参加条件チェック4項目＋その他参加資格

**カテゴリ2: スケジュール（schedule）**

| 抽出項目 | 型 | 説明 | 根拠ソース | assertion_type 例 |
|---------|-----|------|----------|-----------------|
| spec_meeting_date | TIMESTAMP | 仕様説明会日時 | 公告HTML | fact |
| submission_deadline | TIMESTAMP | 入札書提出期限 | 公告HTML | fact |
| opening_date | TIMESTAMP | 開札日時 | 公告HTML | fact |
| equivalent_deadline | TIMESTAMP | 同等品申請期限（物品案件） | 公告HTML | fact |
| quote_deadline | TIMESTAMP | 下見積もり提出期限 | 仕様書PDF | fact / inferred |
| performance_deadline | TIMESTAMP | 履行期限 | 公告HTML or 仕様書PDF | fact |

> **ナレッジ参照**: KNOWLEDGE_DIGEST STEP2 のスケジュール項目

**カテゴリ3: 業務内容（business_content）**

| 抽出項目 | 型 | 説明 | 根拠ソース | assertion_type 例 |
|---------|-----|------|----------|-----------------|
| business_type | VARCHAR | 業務種別（物品の販売/役務の提供） | 仕様書PDF | fact |
| summary | TEXT | 業務内容の要約（200字以内） | 仕様書PDF | inferred（要約のため） |
| items | JSONB | 品目リスト（物品）or 作業内容リスト（役務） | 仕様書PDF | fact |
| delivery_locations | JSONB | 納入場所/作業場所（複数可） | 仕様書PDF | fact |
| contract_type | VARCHAR | 契約種別（スポット/年間） | 仕様書PDF | fact / inferred |
| has_quote_requirement | BOOLEAN | 下見積もり書の要否 | 仕様書PDF | fact |
| has_spec_meeting | BOOLEAN | 仕様説明会の有無 | 公告HTML | fact |

> **ナレッジ参照**: KNOWLEDGE_DIGEST STEP3 の業務内容確認ポイント（物品/役務それぞれ）

**カテゴリ4: 提出物（submission_items）**

| 抽出項目 | 型 | 説明 | 根拠ソース | assertion_type 例 |
|---------|-----|------|----------|-----------------|
| bid_time_items | JSONB | 入札参加時の提出物リスト | 公告HTML + 仕様書PDF | fact |
| performance_time_items | JSONB | 履行時の提出物リスト | 仕様書PDF | fact / inferred |

各提出物の構造:
```json
{
  "name": "入札書",
  "template_source": "発注機関指定書式",
  "deadline": "2026-03-15",
  "notes": "機関の書式以外は無効。郵送の場合は書留 or レターパック",
  "assertion_type": "fact"
}
```

> **重要**: 入札参加時と履行時を明確に区別する（KNOWLEDGE_DIGEST STEP3: 「提出物の確認」）
> **注意**: 下見積もり書は入札書提出前に提出が必要。金額は入札参加額より高い金額を記入すること

**カテゴリ5: リスク要因（risk_factors）**

| リスクタイプ | 検出条件 | 深刻度 | assertion_type |
|------------|---------|-------|---------------|
| 下見積もり期限切れ | quote_deadline < 現在日 | 高 | fact（日付比較） |
| 下見積もり期限迫り | quote_deadline - 現在日 < 3日 | 中 | fact（日付比較） |
| 複数納入先 | delivery_locations の件数 > 1 | 中 | fact |
| 特殊許認可要件 | additional_requirements に許認可記載 | 高 | fact / caution |
| Pマーク/ISO要件 | additional_requirements に ISO/Pマーク記載 | 高 | fact |
| 実績要件あり | additional_requirements に実績要件記載 | 中 | fact / caution |
| 総合評価方式 | 入札方式が総合評価落札方式 | 中 | fact |
| 仕様説明会必須 | 仕様説明会の参加が入札参加条件 | 中 | fact / inferred |

> **assertion_type の使い分け**: リスク検出はルールベースだが、LLMが曖昧な記載から推定した場合は `inferred`、注意喚起として追加した場合は `caution` を付与

#### 3-B-3. プロンプト設計方針

**プロンプトテンプレートの構造:**

```
[システムプロンプト]
- あなたは官公庁入札の公告・仕様書を読解するAIアシスタントです
- 抽出結果は JSON 形式で返してください
- 各項目に以下を付与してください:
  - 根拠情報（source_type, page/selector, section/heading_path, quote）
  - assertion_type（"fact" = テキストに明記, "inferred" = 推定, "caution" = 注意喚起）
- テキストに明記されている情報は assertion_type = "fact" としてください
- テキストから推測・推定した情報は assertion_type = "inferred" とし、推定の理由を付記してください
- テキストに記載がない項目は null とし、推測で値を埋めないでください
- 不明瞭な記載は assertion_type = "caution" とフラグを立ててください

[ユーザープロンプト]
- 以下の公告テキスト・仕様書テキストから、指定された5カテゴリの情報を抽出してください
- [公告テキスト]: {notice_text}
- [仕様書テキスト]: {spec_text}
- [出力JSONスキーマ]: {output_schema}
```

> [要確認] プロンプトの最終設計はPhase0（M6）のPoC時に実際のデータで検証・調整する

#### 3-C-1. 案件カード（case_cards テーブル）

| カラム | 型 | 説明 | 備考 |

| eligibility | JSONB | 参加条件の構造化データ | §3-B-2 カテゴリ1 |

## 4. 非機能要件

### 4-1. 精度指標（2軸: Recall優先 ＋ 根拠付与率）

> **基本方針**: 入札では「誤り」より「抜け」が致命傷。Precision < Recall の方針が安全。

**軸1: フィールド単位のRecall（再現率）**

| 分類 | 対象フィールド | 目標Recall | 根拠 |
|------|-------------|-----------|------|
| **重要項目** | submission_deadline, performance_deadline, unified_qualification, grade, region, bid_time_items, performance_time_items, 無効条件 | **90%以上** | 抜けると参加不可 or 失格リスク |
| **全項目** | 5カテゴリの全抽出フィールド | **80%以上** | 全体の品質基準 |

- 測定方法: Phase0で実際の公告/仕様書10件を人間が正解ラベル付け → LLM抽出結果と比較
- Recall = 正解ラベルのうちLLMが抽出できた項目数 / 正解ラベルの全項目数

**軸2: 根拠付与率**

| 分類 | 対象 | 目標 |
|------|------|------|
| **重要項目** | 上記の重要項目 | **95%以上**（根拠情報 page/section/quote が付与されている割合） |
| **全項目** | 5カテゴリの全抽出フィールド | **85%以上** |

### 4-2. パフォーマンス・非機能要件

| カテゴリ | 要件 | 目標値 |
|---------|------|-------|
| AI読解（1案件） | 公告＋仕様書の読解完了 | **p95: 30秒以内、p99: 60秒以内**（LLM API呼び出し含む） |
| キャッシュ | 同一PDF（SHA-256一致）の再読解 | キャッシュヒット時: 2秒以内（LLM API呼び出しスキップ） |
| セクション分割 | 長文仕様書（5,000トークン超）の処理 | チャンク分割→並列抽出→マージ。p95内に収める |
| PDF対応 | テキスト抽出可能なPDF | Phase1はテキストPDFのみ。スキャンPDFは `status='failed'` + `is_scanned=true` |
| 根拠照合精度 | evidence と元テキストの照合成功率 | 90%以上 |
| LLM APIコスト | 1案件あたりのAPI費用 | [要確認] Phase0で計測 |
| 同時処理 | バッチでの同時読解数 | Phase1: 逐次処理（1案件ずつ） |
| ストレージ | 原本HTML/PDF + 抽出テキスト + 案件カード | 数MB/案件（PDF含む） |
| 可用性 | LLM API障害時 | status='failed' で記録。手動再実行可能 |
| 監視 | 読解の成否・信頼度 | ログ＋低信頼度アラート＋assertion_counts集計 |

  10. レスポンスJSONバリデーション（pydantic スキーマ + assertion_type 検証）

Stage 3 品質管理:
  11. 根拠マッピング・局所探索照合ロジック（PDF/HTML固定構造）
  12. 信頼度スコア算出 + assertion_counts 集計
  13. risk_level 自動算出（high/medium/low）
  14. needs_review 判定・アラート

連携:
  15. F-003 向けデータ提供I/F
  16. F-004 向けデータ提供I/F（チェックリスト自動生成の入力として明示）
  17. 読解トリガー（F-001ステータス変更 → 自動実行）
```

### 6-4. 将来拡張

- **画像PDF対応（Phase2）**: OCR（Tesseract / Google Vision API）フォールバックを有効化。`extraction_method='ocr'`、`confidence_cap=0.7` で品質管理
- **プロンプト学習**: Phase2以降、抽出結果の人間修正データを蓄積 → プロンプト改善
- **マルチLLM対応**: Claude 以外のLLM（GPT-4o等）も選択可能に。コスト・精度で使い分け
- **用語辞書の自動拡張**: 新しい専門用語を検出 → ユーザー確認 → 辞書に追加
- **類似案件のパターン学習**: 同じ発注機関・同じ業務種別の過去案件カードを参照し、抽出精度を向上
