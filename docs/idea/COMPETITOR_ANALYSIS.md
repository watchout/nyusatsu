#### 基本情報

| 項目 | 内容 |
|------|------|

#### ポジショニング

| 項目 | 内容 |
|------|------|
| ターゲット | 入札担当者全般（中小〜大企業） |
| 価値提案 | 官公庁・公的機関の入札情報を網羅的に収集し、検索・管理を効率化 |
| 主な訴求ポイント | 全国の入札情報を一元検索、AIレコメンド、案件管理機能 |

| プラン | 価格 | 内容 |
|-------|------|------|

#### 機能比較

| 機能 | 有無 | 評価 | コメント |

| 落札情報検索 | ✅ | ★★★ | 過去の落札実績検索が充実 |
| 仕様書AI読解 | ❌ | — | 未対応（検索はできるが中身の理解は人手） |
| 入札書類の自動生成 | ❌ | — | 未対応 |
| 参加可否の自動判定 | ❌ | — | 未対応 |

#### ユーザーレビュー分析

**良い点（よく言及される）**:
- 全国の案件を一箇所で検索できる
- 検索キーワードに基づく新着通知が便利
- 落札情報で相場感がつかめる

**悪い点（よく言及される）**:
- 月額料金が高い（特に小規模事業者にとって）
- 探すところまでで終わり。実務は自分でやる必要がある
- UIが情報過多で初心者には取っつきにくい

#### 強み・弱み

| 強み | 弱み |
|------|------|
| 圧倒的なデータ収集量・網羅性 | 「探す」以降の実務支援がない |
| 上場企業の信頼性・継続性 | 価格が中小・個人には高い |
| AIレコメンドで発見力がある | 入札初心者向けのガイド機能が弱い |

| 項目 | 内容 |

#### なぜ間接競合か
- ターゲット業種（建設・土木）が異なる
- ただし「仕様書の中身検索」という思想は入札ラクダAIの要件（仕様理解の自動化）と共通する
- 将来的に物品・役務領域で同様のアプローチを取ってくる可能性あり

#### 学べるポイント
- **付随資料テキスト検索**: 公告だけでなく仕様書・別紙のテキストを解析して検索可能にする発想。入札ラクダAIではこれをAI読解まで拡張する

## 3. 代替手段の分析

### 3.1 現状の解決方法

| 代替手段 | 利用者 | 利点 | 欠点 |
|---------|-------|------|------|
| ホットプラス（入札コンサル） | 入札初心者〜中級者 | ノウハウ提供・個別サポート・データウェアネット | 月額コストが重い。実行は全て自分。スケールしない |
| Excel＋手作業 | 全ての入札事業者 | 無料、柔軟、慣れている | 手動作業が膨大。案件漏れ・期限切れのリスク |
| 調達ポータル直接利用 | 官公庁に慣れた担当者 | 公式データ、無料 | UIが使いにくい。検索・絞り込みが煩雑 |
| データウェアネット | ホットプラス会員 | 案件検索＋CSV出力 | UIが使いにくい。10,000件制限。探すだけで疲れる |
| 何もしない（統一資格の放置） | 資格取得後に止まった人 | ゼロコスト | 入札事業が全く立ち上がらない。機会損失が最大 |

### 3.2 代替手段からの移行障壁

| 障壁 | 対策 |
|------|------|
| 学習コスト | ホットプラスのノウハウをAIに吸収させ、ガイド付きで進行。初心者でも迷わない設計 |
| データ移行 | Phase1は自社利用のため移行不要。Phase2では過去案件のCSVインポート機能を用意 |
| 既存ワークフローの変更 | 「追加」ではなく「置き換え」。案件探索〜準備の全工程をカバーするため、部分導入の煩雑さがない |
| コンサル依存からの脱却心理 | 「コンサルのノウハウがAIに入っている」安心感。初回は無料で1件の成功体験を提供 |

## 6. 差別化ポイント

### 6.1 自社の強み（競合に対して）

| # | 強み | 詳細 | 対象競合 |
|---|------|------|---------|
| 1 | 「探す→読む→準備する」まで一気通貫 | 既存サービスは「探すまで」で止まる。入札ラクダAIは仕様書AI読解→参加可否判定→チェックリスト生成→応札準備まで半自動化 | 全競合 |
| 2 | 仕様書の「理解」を標準機能にする | 用語解説・要点抽出・参加条件チェック・提出物リスト化で、初心者の「何をすればいいかわからない」を直接潰す | NJSS, 入札王 |
| 3 | 公式データ×自社データで勝ちパターン学習 | 調達ポータルの落札実績オープンデータ＋自社の落札/失注データを蓄積し、スコアリングが使うほど賢くなる | 全競合 |
| 4 | 自分がユーザー＝最速フィードバック | Phase1は自分自身が唯一のユーザー。毎日使って改善ループが最速で回る | 全競合 |

### 6.2 自社の弱み（競合に対して）

| # | 弱み | 詳細 | 対策 |
|---|------|------|------|
| 1 | データ収集の網羅性 | NJSSは数年のクローリング実績。入札ラクダAIはゼロからの構築 | 調達ポータルオープンデータ＋公開入札サイトの段階的クロール。まず対象3領域に集中 |
| 2 | ブランド・信頼性がない | NJSS=上場企業、入札王=老舗。入札ラクダAIは無名 | Phase1は自社利用で実績づくり。Phase2はホットプラスコミュニティ経由で口コミ |
| 3 | AI読解の精度リスク | 仕様書の誤読→無効入札・機会損失の可能性 | 必ず原文リンク＋根拠箇所を保持。要約は断定しない設計。人間の最終確認を前提 |
| 4 | 1人開発の限界 | 開発速度・保守・障害対応がボトルネックになる可能性 | MVP機能を絞り、Phase1は自分だけが使う前提で品質基準を現実的に設定 |

### 6.3 差別化戦略

```
■ 基本方針: 「入札情報サービス」ではなく「入札オペレーションOS」

  NJSS/入札王 = 「見つける（探索）」が主戦場
  入札ラクダAI = 「進める（実務）」が主戦場

  → NJSSと「安い/高い」で比較される土俵に乗らない
  → NJSSの後工程を奪う/補完する別カテゴリとして位置づける
  → ホットプラス（人手サービス）のノウハウをツール化して代替する

■ 価格戦略: 低価格で勝負しない

  安さではなく「費用対効果（時短 × リスク低減 × 勝率向上）」で値付け。
  業務の中核に入り込み、工数と無効リスクを削る＝解約されにくいポジション。

  Phase2 プラン設計:
  - Starter: 案件カード＋根拠提示＋チェックリスト（MVPの延長）
  - Pro（本命）: 失注学習ループ＋テンプレ生成＋外注/協力会社連携
  - Biz（B向け）: 会員テンプレ＋横断ダッシュボード＋監査/権限

■ 差別化の3軸:

  1. 「入札オペレーションOS」として実務自動化まで踏み込む
     → NJSSや入札王が手を出していない「読む→準備する→学習する」領域に集中
     → 「探す」では勝てないので、探した後の価値で勝負

  2. 初心者に特化した「わかる」体験を提供
     → ホットプラスのノウハウをAIに吸収させ、仕様書を読むたびに
       用語解説・参加条件チェック・リスク指摘が自動で出る
     → ペルソナ2（佐藤さん）が「コンサルなしで初めて落札できた」と言える体験

  3. データ蓄積による勝ちパターン学習で参入障壁を築く
     → 使うほど落札/失注データが溜まり、スコアリング精度が上がる
     → 後発が追いつきにくい「独自データ資産」になる
```

| 仕様書のAI読解＋要点抽出 | LLM活用の技術的ハードル＋精度保証の難しさ | ✅ LLM＋根拠保持で実現可能。精度は人間確認前提で段階的に向上 |
| 参加条件の自動判定 | ルールが複雑＋ユーザーの資格情報が必要 | ✅ ルールベース（統一資格・等級・地域）でMVP実装可能 |
| 落札/失注の学習ループ | ユーザーの結果データを集める仕組みがない | ✅ Phase1は自社データ。Phase2でユーザーデータ蓄積 |
| 下請けマッチング＋案件アサイン | 入札情報サービスのスコープ外 | ✅ Phase1は自社NW（義父:軽運送、知人:内装・電気）で実現 |
| 入札書・封筒テンプレの自動生成 | 手続きの自動化は入札情報SaaSの範囲外 | ✅ ホットプラスのノウハウ（書式・封筒ルール）をテンプレ化 |

### 8.3 避けるべきこと

| 避けること | 理由 |
|-----------|------|
| NJSSと「データ収集量」で正面勝負 | 上場企業の数年分の蓄積に1人では勝てない。対象3領域に集中する |
| NJSSと「安い/高い」の価格競争に入る | カテゴリが違う（情報サービス vs オペレーションOS）。費用対効果で勝負する |
| Phase1から全機能を作り込む | 使わない機能を作るリスク。自分が毎日使う機能だけにMVPを絞る |
| AI読解の結果を「確定情報」として提示 | 誤読リスクがある。必ず「根拠箇所＋原文リンク」をセットで提示し、最終判断は人間に委ねる |
| 電子入札の提出行為そのものを自動化 | 認証・規約・運用リスクが大きい。まずは「準備の自動化」に徹する |

## 9. 結論

### 9.1 競争環境の評価

| 観点 | 評価 | コメント |
|------|------|---------|
| 競争の激しさ | 中 | 「探す」領域はNJSS/入札王が強い。「実務自動化」領域は未開拓 |
| 参入障壁 | 中 | データ収集にコストがかかるが、公式オープンデータと対象領域の絞り込みで現実的 |
| 差別化の余地 | 高 | 「探す→読む→準備する→学習する」の一気通貫は競合が手を出していない |
| 勝てる可能性 | 高 | Phase1は自分だけが使う。競合との勝負ではなく「自分の入札事業を回せるか」が成功基準 |

### 9.2 競争戦略サマリー

```
■ カテゴリ定義: 「入札情報サービス」ではなく「入札オペレーションOS」

  NJSS/入札王と同じ土俵で戦わない。
  彼らの「後工程」を奪う/補完する別カテゴリとして参入する。

■ 勝ち筋: 「探した後」で勝つ

  案件を見つけた後の「読む→判断する→準備する→学習する」を半自動化し、
  入札初心者でも元請けとして事業を回せるOS。

■ 価格: 低価格で勝負しない

  業務の中核に入り込み、費用対効果（時短×リスク低減×勝率）で値付け。
  Phase1のMVPで成果指標を計測してからPhase2の料金を設計する。

■ Phase展開:
  Phase1: 自分が毎日使って実績を作る
  Phase2: 同じ課題を持つ中小事業者にSaaS展開（Starter/Pro/Biz）
  Phase3: コンサル会社（ホットプラス等）とJV（4:6按分）
```
