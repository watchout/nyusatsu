## 2. ユーザーストーリー

### メインストーリー

> **ペルソナ1（金子）として、**
> 毎朝10分の案件チェックで、対象3領域（軽運送・物品・内装）の
> スコア付き案件一覧を確認できるようにしたい。
> 「いま参加すべき案件はこれ」がひと目でわかれば、
> 本業が忙しくても入札事業を回せる。

### 具体シナリオ

**シナリオ1: 朝のダッシュボード確認**
1. 日次バッチが深夜に実行され、新着案件を収集済み
2. 金子が朝ダッシュボードを開く
3. 新着案件がスコア順に表示される（スコア85「○○省 軽運送業務」等）
4. スコアの根拠が表示される：「競争少（平均2.3社）・相場120万円・期限余裕あり」
5. → 上位案件から詳細を確認し、応札するか判断する

**シナリオ2: 検索条件の調整**
1. デフォルトの3領域に加えて「電気工事」を追加したい
2. 設定画面でキーワードに「電気工事」を追加
3. 次回バッチで「電気工事」関連の案件も収集・スコアリングされる
4. → 事業拡大に合わせて対象領域を柔軟に広げられる

**シナリオ3: 新着アラート**
1. スコア70以上の新着案件が検出された
2. アラート通知（Phase1はログ出力。将来的にメール/Slack）
3. → 重要案件を見逃さない

**シナリオ4: 案件のステータス管理**
1. 案件一覧で「応札予定」に変更 → F-002のAI読解に送る
2. 「見送り」に変更 → 理由をメモ付きで記録
3. 受付終了した案件は自動で「アーカイブ」に移動
4. → 案件パイプラインとして管理できる

## 3. 機能要件

### 3-A. 入力

#### 3-A-1. データソース（複数ソース並行）

| # | データソース | 形式 | 取得方法 | Phase1対象 | 取得内容 | 備考 |
|---|------------|------|---------|-----------|---------|------|
| 1 | 調達ポータル 入札公告情報 | HTML | スクレイピング | ○（優先） | 案件名、発注機関、入札方式、提出期限、開札日、仕様書URL、公告URL | 入札受付中の案件を取得。メインのデータソース |
| 2 | 調達ポータル 落札実績OD | CSV | F-005 base_bids と共有 | ○ | 過去案件情報 → スコアリング参照 | F-005が取り込んだbase_bids/bid_detailsをJOINしてスコアリングに利用 |
| 3 | 官公庁個別サイト | HTML | スクレイピング | △（Phase1後半） | 省庁固有の公告ページ | 調達ポータルに載らない案件をカバー。段階的に追加 |

```
データソース1: 調達ポータル入札公告（メイン）
       ↓ 新着案件
データソース2: F-005 base_bids/bid_details（参照）
       ↓ 相場データ
       ↓
  → 正規化 → フィルタ → スコアリング → 案件カード
       ↑
データソース3: 官公庁個別サイト（段階的追加）
```

#### 3-A-2. ユーザー入力（設定）

| 項目 | 内容 | デフォルト | 変更頻度 |
|------|------|----------|---------|
| 対象領域キーワード | 収集対象の案件を特定するキーワード群 | 「軽運送」「配送」「物品」「清掃」「内装」 | 低（初期設定後は稀に変更） |
| 対象地域 | 競争参加地域 | 全国 | 低 |
| 対象等級 | 自身の資格等級 | D等級 | 低 |
| 入札方式フィルタ | 一般競争入札のみ/指名含む | 一般競争入札のみ | 低 |
| スコア閾値（アラート） | アラートを出す最低スコア | 70 | 中 |
| バッチ実行時刻 | 日次バッチの実行時刻 | 04:00 | 低 |

### 3-B. 処理

#### 3-B-1. データ取得パイプライン（アダプターパターン）

```
═══════════════════════════════════════════════
  共通パイプライン（全データソース共通）
═══════════════════════════════════════════════

[データソースアダプター]
       │
       │  各アダプターが統一インターフェースを実装
       │  → fetch() → normalize() → store()
       │
       ▼
  ① fetch(): データソースからデータ取得
       │  → HTML/CSV/JSON をアダプター固有のロジックで取得
       │
       ▼
  ② normalize(): 統一案件スキーマに変換
       │  → 各データソース固有の項目を cases テーブルの形式に正規化
       │
       ▼
  ③ store(): DB格納（重複排除＋差分検知）
       │  → source + source_id をキーに UPSERT
       │  → 新規/更新/変更なし を検知・記録
       │
       ▼
  ④ filter(): フィルタリング
       │  → キーワードマッチ（対象領域）
       │  → 条件フィルタ（地域、等級、入札方式）
       │  → 期限フィルタ（受付終了案件を除外）
       │
       ▼
  ⑤ score(): スコアリング
       │  → F-005 base_bids/bid_details を参照
       │  → 4要素でスコア算出（0-100）
       │
       ▼
  ⑥ notify(): アラート
       → スコア閾値以上の新着案件をアラート
```

**アダプター設計:**

| アダプター | クラス名 | 対象 | Phase1実装 |
|-----------|---------|------|-----------|
| 調達ポータル入札公告 | `ChotatkuPortalAdapter` | 調達ポータルの入札公告ページ | ○（最優先） |
| 官公庁個別サイト | `GovSiteAdapter` | 省庁固有の公告ページ | △（Phase1後半） |

**共通インターフェース:**

```
class BaseSourceAdapter:
    def fetch() -> list[RawCase]          # データソースからデータ取得
    def normalize(raw: RawCase) -> Case   # 統一スキーマに変換
    def store(cases: list[Case]) -> StoreResult  # DB格納
```

> **設計意図**: 新しいデータソースを追加する場合、BaseSourceAdapter を継承した
> 新アダプターを作成するだけで対応可能。既存のフィルタリング・スコアリングは再利用できる。

**アダプター1: 調達ポータル入札公告（ChotatkuPortalAdapter）**

| 項目 | ルール |
|------|-------|
| 取得対象 | 入札受付中の公告（一般競争入札） |
| 取得方法 | [要確認] HTML スクレイピング or API |
| ページネーション | [要確認] 公告一覧の構造に依存 |
| 取得内容 | 案件名、発注機関、入札方式、提出期限、開札日、仕様書URL、公告詳細URL |
| アクセス間隔 | 1リクエストあたり最低1秒（サーバー負荷配慮） |
| リトライ | 取得失敗時、最大3回リトライ（間隔: 30秒→60秒→120秒） |
| 失敗時対応 | 3回失敗でアラート。次回バッチで再試行 |
| 実行頻度 | 日次（デフォルト） |
| 原本保存 | 取得したHTMLを `data/raw/portal/` にタイムスタンプ付きで保存 |

**アダプター2: 官公庁個別サイト（GovSiteAdapter）— Phase1後半**

| 項目 | ルール |
|------|-------|
| 対象省庁 | [要確認] Phase1で対象とする省庁の優先順位 |
| 取得方法 | 省庁ごとのHTMLスクレイピング |
| 特性 | 省庁ごとにページ構造が異なるため、個別パーサーが必要 |
| 優先度 | 調達ポータルでカバーできない案件がある場合に段階的に追加 |

#### 3-B-2. 正規化・格納

**テーブル: cases（統一案件スキーマ）**

| カラム | 型 | 説明 | 備考 |
|--------|-----|------|------|
| id | UUID | 内部ID | 自動採番（PK） |
| source | VARCHAR | データソース識別子 | 「chotatku_portal」「gov_xxx」等 |
| source_id | VARCHAR | データソースでの案件識別子 | (source, source_id) でUNIQUE |
| case_name | TEXT | 案件名 | |
| issuing_org | VARCHAR | 発注機関名 | |
| issuing_org_code | VARCHAR | 発注機関コード | 取得できない場合はNULL |
| bid_type | VARCHAR | 入札方式（一般競争入札/指名競争入札等） | |
| category | VARCHAR | 品目分類（物品/役務等） | |
| region | VARCHAR | 競争参加地域 | |
| grade | VARCHAR | 等級（A/B/C/D） | |
| submission_deadline | TIMESTAMP | 入札書提出期限 | |
| opening_date | TIMESTAMP | 開札日時 | |
| spec_url | TEXT | 仕様書URL | |
| notice_url | TEXT | 公告URL | |
| detail_url | TEXT | 公告詳細ページURL | |
| status | VARCHAR | ステータス | new/reviewed/planned/skipped/archived |
| skip_reason | TEXT | 見送り理由 | status=skipped の場合 |
| score | INTEGER | スコア（0-100） | スコアリング後に更新 |
| score_detail | JSONB | スコアの内訳 | `{"competition": 25, "scale": 30, ...}` |
| raw_data | JSONB | 取得した元データ全体 | スキーマ変更への備え |
| first_seen_at | TIMESTAMP | 初回検出日時 | |
| last_updated_at | TIMESTAMP | 最終更新日時 | |
| archived_at | TIMESTAMP | アーカイブ日時 | 受付終了時に自動セット |

**DB: PostgreSQL（F-005で確定済み）**

**重複排除ルール:**
- `(source, source_id)` の複合ユニークキーで重複チェック
- 同一キーが存在する場合、変更があればUPSERT（案件名・期限等の更新）
- 変更がなければスキップ（last_updated_at のみ更新）
- 新規レコードは `status = 'new'` で挿入

**差分検知:**
- 前回バッチとの差分を検知: 新規（new） / 更新（updated） / 変更なし（unchanged）
- 差分結果は `batch_logs` テーブルに記録

**テーブル: batch_logs（バッチ実行ログ）**

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | PK |
| source | VARCHAR | データソース識別子 |
| started_at | TIMESTAMP | 実行開始日時 |
| finished_at | TIMESTAMP | 実行終了日時 |
| status | VARCHAR | success/failed/partial |
| total_fetched | INTEGER | 取得件数 |
| new_count | INTEGER | 新規件数 |
| updated_count | INTEGER | 更新件数 |
| unchanged_count | INTEGER | 変更なし件数 |
| error_count | INTEGER | エラー件数 |
| error_details | JSONB | エラー詳細 |

#### 3-B-3. フィルタリング

案件がユーザーの対象範囲に合致するかを判定する。

**フィルタリングルール:**

| # | フィルタ | 条件 | 動作 |
|---|--------|------|------|
| 1 | キーワードマッチ | `case_name` に対象領域キーワードのいずれかを含む | OR条件で部分一致 |
| 2 | 入札方式 | ユーザー設定の入札方式に合致 | デフォルト: 一般競争入札のみ |
| 3 | 地域 | ユーザー設定の対象地域に合致（or 全国） | 全国指定時はフィルタなし |
| 4 | 等級 | ユーザー設定の等級以上 | D等級は全てパス |
| 5 | 期限 | 提出期限が現在日時より後 | 受付終了案件を除外 |

**フィルタ結果:**
- フィルタを通過した案件のみスコアリングに進む
- フィルタで除外された案件もDBには保存（raw_data として）。再フィルタ可能

#### 3-B-4. スコアリング

F-005の相場データを参照し、案件の「おいしさ」を数値化する。

**スコア構成（0-100点）:**

| # | 要素 | 配点 | 算出方法 | データソース |
|---|------|------|---------|------------|
| 1 | 競争度 | 30点 | 同領域の平均参加社数が少ないほど高得点 | F-005 bid_details.num_participants |
| 2 | 案件規模 | 25点 | 落札金額の中央値が適正範囲内で高得点 | F-005 base_bids.winning_amount |
| 3 | 期限余裕 | 25点 | 提出期限までの日数が長いほど高得点 | cases.submission_deadline |
| 4 | 領域適合度 | 20点 | 対象キーワードとの一致度（完全一致>部分一致） | cases.case_name |

**各要素の詳細:**

**① 競争度（30点満点）:**

| 平均参加社数 | 得点 |
|------------|------|
| 1社（独占） | 30 |
| 2社 | 25 |
| 3社 | 20 |
| 4〜5社 | 15 |
| 6〜10社 | 10 |
| 11社以上 | 5 |
| データなし | 15（中央値） |

> **注**: 参加社数は同一発注機関×同一領域のbid_details.num_participantsの平均を使用。
> データがない場合はデフォルト15点。

**② 案件規模（25点満点）:**

| 落札金額中央値（同領域） | 得点 |
|------------------------|------|
| 50万〜500万円（適正範囲） | 25 |
| 500万〜1,000万円 | 20 |
| 20万〜50万円 | 15 |
| 1,000万円以上 | 10 |
| 20万円未満 | 5 |
| データなし | 15（中央値） |

> **注**: 「適正範囲」はPhase1開始後に実績ベースで調整する。
> 初期値はユーザー（金子）の事業規模に基づく仮値。

**③ 期限余裕（25点満点）:**

| 提出期限までの日数 | 得点 |
|------------------|------|
| 14日以上 | 25 |
| 10〜13日 | 20 |
| 7〜9日 | 15 |
| 4〜6日 | 10 |
| 1〜3日 | 5 |
| 当日 or 超過 | 0 |

**④ 領域適合度（20点満点）:**

| 一致度 | 得点 |
|-------|------|
| キーワード完全一致（案件名に「軽運送」が含まれる等） | 20 |
| 関連キーワード一致（「配送」「運送」等の類義語） | 15 |
| 部分一致（「運」「送」が含まれる等） | 10 |
| カテゴリのみ一致 | 5 |

**スコアリングの更新タイミング:**
- 案件が新規追加された時
- F-005のデータが更新された時（相場の変動を反映）
- ユーザーが検索条件を変更した時

[要確認] スコアリングの重み付け（各要素の配点比率）は、Phase1運用後に実績データに基づいて調整する予定。初期値は仮設定。

### 3-D. ビジネスルール

| # | ルール | 詳細 |
|---|-------|------|
| 1 | 原本保存 | 取得した公告HTMLを `data/raw/portal/` にタイムスタンプ付きで保存。削除は手動のみ |
| 2 | 差分検知 | 同一案件の変更（期限変更、内容修正等）を検知し、更新日時とともに記録 |
| 3 | 自動アーカイブ | 提出期限を過ぎた案件は `status = 'archived'` に自動変更。日次バッチで実行 |
| 4 | スコア再計算 | F-005のデータ更新時、対象領域の全アクティブ案件のスコアを再計算 |
| 5 | データソース障害時の継続 | 1つのデータソースが障害でも、他のソースは正常に実行。部分的な成功を許容 |
| 6 | アクセスマナー | 1リクエスト/秒以下のレート制限。robots.txt を尊重 |
| 7 | ステータス遷移 | new → reviewed → planned/skipped → archived の順序。巻き戻しは手動のみ |
| 8 | キーワード辞書 | 対象領域キーワードは将来的に類義語辞書に拡張可能な設計とする |

**ステータス遷移図:**

```
  ┌──────┐
  │ new  │ ← 初回検出
  └──┬───┘
     │ ユーザーが確認
     ▼
  ┌──────────┐
  │ reviewed │ ← 確認済み
  └──┬───┬───┘
     │   │
     ▼   ▼
┌────────┐ ┌─────────┐
│planned │ │ skipped │ ← 見送り（理由メモ付き）
└──┬─────┘ └────┬────┘
   │            │
```

### 6-2. 技術選択の方針

| 項目 | 方針 | 備考 |
|------|------|------|
| DB | **PostgreSQL（F-005で確定済み）** | casesテーブルもF-005と同一DBインスタンス |
| バッチ実行 | Python スクリプト + cron（Phase1） | 後にCelery等に移行可能 |
| HTTP取得 | httpx（非同期対応） | リトライは tenacity で実装 |
| HTMLパース | beautifulsoup4 | F-005 Layer 2 と共通 |
| スコアリング | SQLクエリ + Python | F-005のテーブルをJOINして集計値取得 → Pythonでスコア算出 |
| アラート | ログ出力（Phase1） | 将来的にメール/Slack連携 |

### 6-3. 開発順序（F-001内部）

```
基盤:
  1. cases テーブル + batch_logs テーブル定義
  2. BaseSourceAdapter インターフェース設計
  3. 正規化・格納ロジック（UPSERT + 差分検知）

データ取得:
  4. ChotatkuPortalAdapter 実装
     4a. 公告一覧ページのパーサー
     4b. 公告詳細ページのパーサー
     4c. 原本保存ロジック
  5. バッチ実行スクリプト（cron登録）

フィルタリング・スコアリング:
  6. フィルタリングロジック（キーワード + 条件）
  7. スコアリングロジック（F-005データ参照）
  8. スコア再計算バッチ

出力・管理:
  9. 案件カード一覧API（JSON）
  10. ステータス管理API
  11. バッチサマリー・アラートロジック

Phase1後半:

```
